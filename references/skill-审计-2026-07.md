### P1-6（新发现，由 P2-4 自检脚本首次运行时自动捕获）：project-proposal-generator 与 docs-archive-manager SKILL.md frontmatter YAML 解析失败  ✅ 已修复 (2026-07-19)

**证据**（自检脚本输出）：
```
[P1 FAIL] Invalid SKILL.md frontmatter (while scanning a simple key): project-proposal-generator
[P1 FAIL] Invalid SKILL.md frontmatter (mapping values are not allowed here): docs-archive-manager
```

**根因**：
- project-proposal-generator：`description: |-` block 后跟了未缩进的段落（`仅面向内部决策...` 和 `不做的事情：` 列表），YAML 把这些当成新的 top-level key，但缺少 `:` 报错
- docs-archive-manager：`compatibility:` inline value 含 ASCII `: `（`Default docs root: /opt/...`），被 YAML 误解析为嵌套 mapping

**影响**：
1. 严格 YAML parser（含 python `yaml.safe_load`）解析失败
2. OpenCode 的 skill loader 可能用更宽松的 parser 仍能加载，但 description / compatibility 字段会被截断或丢失
3. agent 读 description 时拿到的可能是残缺文本，触发词覆盖下降

**修复**：
- project-proposal-generator：把未缩进段落合并回 `|-` block 内
- docs-archive-manager：把 `compatibility:` 改为 `|-` literal block

**教训**：**人工审计必然漏**——本次审计过了 3 个批次，人工检查都没发现 YAML 解析问题；自检脚本第 0 次运行就抓到了。这正是 P2-4 的核心价值证明。

---

### P1-6 的治理意义

P1-6 是**自检脚本自己发现的**，不是手工审计发现的。这反向证明了 P2-4 的必要性：

> 如果一个治理工具只能检查你**已经知道**的问题，那它只是把人工检查脚本化。
> 真正的治理工具应该能检查你**不知道你不知道**的问题——YAML 合法性就是这种，**没人会主动想到要去 grep 检查 YAML 解析**，但脚本加了一条 `yaml.safe_load` 就抓到了。

后续扩展自检脚本的方向：
- frontmatter 字段名拼写检查（typo detection）
- description 中"触发场景"段落的最低关键词密度
- references/ 中文件的实际可读性（避免引用空文件）
- pyproject.toml 的 dependencies 与 SKILL.md compatibility 字段一致性

## 八、已知限制与后续工作（out of current scope）

### 限制 1：fresh clone 部署无自动化

`.opencode/` 在根 `.gitignore` 内（line 14），所以 git clone 后所有 21 个软链都缺失——首次跑 `check_skill_ecosystem.sh` 会出 21 条 P0 FAIL。

**未来工作**（不在本次审计范围）：

### 限制 2：description 长度阈值是经验值

自检脚本用 200 字符作为 description 警告阈值，这个数字来自 peer skill 中位数观察（约 400），不是 OpenCode 官方规范。某些极简 skill（如 winshang-crawler 31 字符）可能永远达不到阈值，且这是合理的（一个爬虫 skill 确实不需要复杂触发词矩阵）。

未来工作：把阈值做成 per-skill 可配置（在 SKILL.md frontmatter 加 `audit_description_min: 50` 之类的 override）。

### 限制 3：自检脚本不验证运行时

脚本能验证磁盘状态（软链、SKILL.md 解析、gitignore 模式），但**不能验证 OpenCode 运行时是否真的能加载 skill**。理论上 SKILL.md 解析合法但 OpenCode skill loader 因其它原因（如 schema 版本不匹配）拒绝加载。

未来工作：在 OpenSpec change archive 前，跑一遍 `skill(name="<new-skill-name>")` 实际触发，确认 available_skills 列表里真的出现了。
