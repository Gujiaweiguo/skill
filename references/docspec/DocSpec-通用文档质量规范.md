# DocSpec - 通用文档质量规范

## 目标

DocSpec 保证文档不是“写得像”，而是能被使用、复核、追溯和持续维护。

一份合格文档必须满足四个结果：

1. **可追溯**：关键结论能追到来源、证据、截图、代码、需求或用户确认。
2. **可验收**：读者知道什么算完成、通过、失败、待确认。
3. **可复用**：同类文档结构稳定，能被下游 skill、人工评审或 RAG 使用。
4. **可维护**：后续修改知道影响范围，历史假设和风险不被覆盖。

## Doc Brief

写任何正式文档前，先明确：

| 字段 | 要求 |
|---|---|
| 文档类型 | PRD / 方案 / 投标 / 手册 / PPT / Word / skill / 其他 |
| 目标读者 | 管理层、客户、产品、研发、实施、运维、销售、agent |
| 使用场景 | 决策、评审、交付、培训、投标、RAG、后续生成 |
| 输入来源 | 文件、代码、截图、客户需求、竞品、用户口述、公开资料 |
| 输出格式 | Markdown、Word 内容包、PPT YAML、Excel、最终 Office 文件 |
| 边界 | 明确不做什么，不覆盖哪些事项 |
| 验收方式 | 覆盖矩阵、响应矩阵、链接检查、渲染验证、人工确认 |

## 质量门禁

| 门禁 | 通过标准 | 不通过处理 |
|---|---|---|
| 来源门禁 | 关键事实有来源 | 标为 `待确认`，不得写成结论 |
| 结构门禁 | 有稳定目录、章节职责清晰 | 先补结构，再填正文 |
| 覆盖门禁 | 输入需求、评分点或目标能力有矩阵映射 | 输出缺口清单 |
| 术语门禁 | 同一概念只用一个标准术语 | 建术语对照或别名表 |
| 边界门禁 | 不做事项、假设、风险显式列出 | 加入风险与待确认项 |
| 下游门禁 | 下游 skill / 人工使用方式明确 | 增加使用说明或内容包说明 |
| 验收门禁 | 交付前有检查结果 | 不交付最终版，只交付草稿/待确认版 |

## 证据等级

| 等级 | 定义 | 可写成 |
|---|---|---|
| E1 实证 | 代码、运行截图、合同/招标文件、客户确认、正式手册 | 明确结论 |
| E2 强证据 | 多份一致资料、可复核竞品手册、已交付模板 | 高置信判断 |
| E3 弱证据 | 单份资料、销售话术、截图片段、公开营销页 | 候选判断 |
| E4 推断 | agent 根据上下文推理，无直接证据 | 假设/待确认 |

红线：E3/E4 不能写成“已具备”“已证明”“客户确认”“行业标配”。

## 状态词

统一使用以下状态词，避免同义词混用：

| 状态 | 含义 |
|---|---|
| `已覆盖` | 有充分证据且满足要求 |
| `部分覆盖` | 有能力但不完整，或需配置/人工补充 |
| `缺失` | 当前未发现能力或材料 |
| `待确认` | 信息不足，需要用户或负责人确认 |
| `不适用` | 需求不属于本产品/项目/文档边界 |
| `风险` | 可能影响验收、合规、报价、交付或决策 |

## 必须输出的辅助结构

按文档类型至少选择一种：

| 场景 | 辅助结构 |
|---|---|
| PRD / 产品规划 | 需求证据表、功能清单、差距分析、review 清单 |
| 方案 / 立项 | 客户画像、素材匹配表、价值主张-证据表 |
| 投标 | 招标要求响应矩阵、资格缺口清单、交付检查清单 |
| 手册 | 操作路径、前置条件、故障处理、知识库版说明 |
| PPT / Word | 内容包、素材引用清单、版式/模板验证说明 |
| skill | 触发/不触发场景、输入输出、流程、失败模式、维护规则 |

## 禁止事项

- 不编造客户确认、资质、案例、报价、人员、合同、功能状态。
- 不把“未发现”写成“不存在”。
- 不删除待确认项来显得完整。
- 不把一次性客户细节沉淀成通用规范。
- 不把业务域知识写进跨 skill 通用规范；业务域知识回到项目目录或 `$LANLNK_BASE`。
- 不把模板约束、渲染坑、格式陷阱只留在对话里；可复用经验必须进入文件。
