# Skill 写作质量规范

## 目标

`SKILL.md` 是 agent 执行任务时的操作规程，不是宣传页。一个好 skill 必须让 agent 知道什么时候触发、读什么、怎么做、产出什么、怎样验收、遇到失败怎么办。

## 必备结构

| 部分 | 要求 |
|---|---|
| frontmatter | `name`、触发描述、兼容性、运行依赖 |
| 目标 | 一句话说明这个 skill 解决什么问题 |
| 适用 / 不适用 | 明确触发边界，避免抢其他 skill 的任务 |
| 输入 | 文件、环境变量、用户参数、上游产物 |
| 输出 | 文件路径、格式、内容包、报告、检查清单 |
| 流程 | P0/P1/P2... 每步输入、动作、输出 |
| 质量门禁 | 引用 DocSpec，并列出本 skill 的专属检查 |
| 失败模式 | 缺资料、依赖不可用、格式错误、权限不足时怎么降级 |
| 维护规则 | 哪些知识写本 skill，哪些 promote 到共享规范 |

## 写作规则

- 用操作性语言，少写愿景和营销表达。
- 每个步骤都写清输入、动作、输出。
- 不把“agent 自行判断”作为唯一规则；给判断标准。
- 不把路径写散，先定义路径变量再引用。
- 不在 `SKILL.md` 内沉淀大量业务域知识；域知识放项目目录或业务资料库。
- 复杂规则拆到 `references/`，`SKILL.md` 只保留入口和关键流程。
- 排障经验写 `references/troubleshooting.md`，不要埋在对话或提交说明里。

## 跨 skill 规则

| 规则影响范围 | 写入位置 |
|---|---|
| 只影响一个 skill | 该 skill 的 `SKILL.md` 或 `references/troubleshooting.md` |
| 影响多个文档类 skill | `/opt/code/skill/references/docspec/` |
| 影响所有 skill / OpenCode 操作 | `/opt/code/skill/AGENTS.md` |
| 只影响某个业务项目 | `$LANLNK_BASE/out/prd/<项目>/域知识.md` 或项目 docs |

## 评审清单

改完 skill 后检查：

- 触发场景和不触发场景是否清楚。
- 是否引用了正确的上游/下游 skill。
- 路径变量是否统一。
- 生成物是否有明确路径和格式。
- 是否说明无法生成正式文件时的降级产物。
- 是否有质量门禁和验证方式。
- 是否有复利回流规则。
