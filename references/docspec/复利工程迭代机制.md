# DocSpec 复利工程迭代机制

## 目标

DocSpec 通过复利工程持续变好：每次 PRD、方案、投标、手册、PPT、Word、skill 编写任务完成后，把已验证的通用经验回流到正确位置。

## 触发方式

用户说以下内容时，加载 `compound-learning`：

- “复利工程”
- “沉淀这次经验”
- “以后这类文档都按这个模式”
- “把这次 PRD/投标/方案/手册/skill 经验沉淀一下”

## 回流判断

| 经验类型 | 写入位置 |
|---|---|
| 影响多个文档类 skill 的质量规则 | `/opt/code/skill/references/docspec/` |
| 只影响单个 skill 的坑或限制 | 该 skill 的 `SKILL.md` 或 `references/troubleshooting.md` |
| 影响所有 skill 或 OpenCode 操作 | `/opt/code/skill/AGENTS.md` |
| 业务域术语或产品口径 | `$LANLNK_BASE/out/prd/<项目>/域知识.md` 或业务资料库 |
| 面向人的 OpenCode 使用方法 | `/opt/code/docs/opencode/` |

## 周期性完善

建议每 2-4 周或累计 3 次同类文档任务后做一次 DocSpec 复盘：

1. 搜索最近新增的 `待确认`、`风险`、`troubleshooting`、`quality`、`review`。
2. 找出重复出现的质量问题。
3. 判断是通用规范、单 skill 规则、还是业务域知识。
4. 更新对应文件。
5. 把新增规则同步到相关 skill 的轻量引用或验收章节。

## 抽象成 skill 的条件

当满足以下条件时，把 DocSpec 抽象为独立 `docspec-reviewer` 或 `doc-quality-reviewer` meta skill：

- 至少被 3 类文档任务使用过。
- 验收清单稳定，没有每次大改。
- 复利工程已沉淀出 5 条以上跨 skill 规则。
- 能定义清楚输入、输出、评分方式和失败模式。

## 未来 skill 草案

```text
name: docspec-reviewer
输入: 文档路径 / 内容包路径 / skill 路径 / 文档类型
输出: 质量评分、红线问题、待确认项、修改建议、是否可交付
不做: 代替业务生成，不自动改正文，除非用户明确要求
```
