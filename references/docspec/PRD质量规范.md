# PRD 质量规范

## 目标

PRD 不是功能堆叠，而是产品决策资产。它必须能回答：现在有什么、客户和竞品要求什么、差距是什么、先做什么、为什么。

## 必备产物

| 产物 | 用途 |
|---|---|
| `产品PRD.md` | 面向产品/研发/交付的主文档 |
| `功能清单.md` | 标准能力视图 |
| `需求清单.md` | 客户/竞品/代码等原始需求归集 |
| `需求证据表.md` | 需求到来源的追溯表 |
| `差距分析.md` | 当前状态、差距、建议动作 |
| `高优先级需求review清单.md` | 必须人工确认的高风险项 |

## 质量规则

1. 当前产品、客户需求、竞品资料、代码事实必须分源，不混成单一事实。
2. 每条关键能力必须有 `状态 + 置信度 + 来源`。
3. `missing` 只表示当前未发现，不等于行业不存在或客户不需要。
4. 多来源冲突先进入差距分析或 review，不直接覆盖。
5. 分期按业务闭环，不按模块平均切片。
6. PRD 到实施的交接边界要清楚：默认输出目标蓝图、差异、分期、验收链路，不直接改业务系统代码。

## 状态与置信度

| 状态 | 使用条件 |
|---|---|
| `existing` | 代码或正式文档证明已具备 |
| `partial` | 有部分能力，但流程、字段、权限、报表、移动端或集成不完整 |
| `missing` | 当前未发现实现或资料 |
| `explicitly-not-do` | 明确不属于当前产品边界 |

| 置信度 | 使用条件 |
|---|---|
| `high` | 多源一致或代码可验证 |
| `medium` | 单源明确但未完全验证 |
| `low` | 来源弱、描述模糊、存在冲突 |

## 红线

- 不把竞品营销材料直接写成行业必须能力。
- 不把客户需求里的笼统愿景拆成确定功能，除非有上下文证据。
- 不把开发中、规划中、第三方能力写成已覆盖。
- 不把待确认项并入最终能力清单。

## 缺口优先级分级

当 PRD 通过 coverage-validate / 差距分析发现"客户/竞品/ontology 有定义，但当前 spec 或代码缺失"的缺口时，按以下口径分级排期：

| 优先级 | 判定条件 | 含义 |
|---|---|---|
| **P0** | ≥2 家客户**独立**提出 + ontology 有定义 + 当前 spec/代码缺失 | 多客户共识 + 业务域一致认可，应优先排入下个迭代 |
| **P1** | 1 家客户提出 + ontology 有定义 + 当前 spec/代码缺失 | 单客户需求但业务上合理，按客户优先级排期 |
| **P2** | 仅 ontology 推断 + 客户/竞品未明确提 + 当前 spec/代码缺失 | 业务域完整性需要，但客户未表达，作为长尾观察项 |

### 分级原则

1. **客户独立**指不同客户的原始需求文档各自提出，不是同一集团子公司重复计数。
2. **ontology 有定义**指该能力在 `business-ontology.yaml` 中已建模为 capability 或 term。ontology 未定义的需求即使多家客户提出，也不自动升 P0，先补 ontology。
3. **spec/代码缺失**包括三种状态：openspec 完全没有 spec、有 spec 但代码未引用、有 spec 有代码但客户证据指向未覆盖的子能力。
4. P0/P1/P2 用于内部排期和交接包，不直接对客户承诺；客户侧承诺以分期方案为准。
5. 这套口径对 `requirement-evaluator`、`competitor-product-analyzer`、`product-prd-generator` 同样适用，三处共用同一份分级表，避免评估结论打架。
