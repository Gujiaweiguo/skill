# 方案与投标文档质量规范

## 覆盖范围

适用于公司介绍、客户方案、立项建议书、战略汇报、投标商务标/技术标、响应表、偏离表和报价说明。

## 公司介绍 / 客户方案

必须建立三层链路：

1. 客户现状：行业、已有系统、关键岗位、当前痛点、采购阶段。
2. 蓝联响应：产品组合、首访切口、能力边界、实施路径。
3. 证明材料：案例、资质、截图、交付方法、团队、服务体系。

质量要求：

- 案例不能跨产品冒用；相关案例只能标为相关证明。
- 客户资产组合路由必须说明为什么选该切口。
- 面客材料不暴露内部不确定表述，但内部草稿必须保留待确认项。
- 不用泛泛的“数字化转型”替代具体业务问题。

## 立项建议书

必须回答：

| 问题 | 要求 |
|---|---|
| 为什么做 | 背景、痛点、机会、风险 |
| 做什么 | 方案范围、产品组合、替代方案 |
| 怎么做 | 里程碑、资源、预算、验收 |
| 值不值得 | 成本、收益、风险、推荐结论 |

红线：不把客户方案写成立项论证，不把销售话术写成 ROI 证明。

## 投标文件

投标文档以响应完整性为第一优先级。

必须产出：

- 招标文件解析。
- 招标要求响应矩阵。
- 资格/材料缺口清单。
- 报价口径与公式检查。
- 交付检查清单。

红线：

- 不得编造资质、人员、业绩、授权、报价。
- 不得遗漏“必须/不得/无效投标/实质性响应/资格审查”。
- 不得残留历史模板里的旧项目名、旧客户名、旧日期。
- 不得在未核对公式时覆盖报价汇总单元格。

## 战略简报

战略文档要分清事实、判断和建议：

| 层级 | 示例 |
|---|---|
| 事实 | 市场趋势、竞品动作、客户 SOP、现有产品能力 |
| 判断 | 机会窗口、差异化定位、路线选择 |
| 建议 | 产品路线、销售切口、试点打法、下一步验证 |

缺资料的判断必须标记假设，不写成结论。
