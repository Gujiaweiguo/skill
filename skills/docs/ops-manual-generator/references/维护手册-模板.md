## 二、日常巡检

### 2.1 每日巡检清单

【标准内容 + 从 healthcheck 读取端点】。运维人员每日执行：

```bash
cd /opt/{系统名}

# 1. 服务状态检查
docker compose ps
# 预期：所有服务 STATUS 为 Up 或 Up (healthy)

# 2. 健康端点检查
curl -sf http://localhost/health | jq .
# 预期：返回 {"status": "ok"} 或 HTTP 200

# 3. 磁盘空间检查
df -h /opt/{系统名}
# 预期：Use% < 80%，超过 85% 需清理（参考 2.3 日志清理）

# 4. 内存检查
free -h
# 预期：available > 总内存的 20%

# 5. 容器资源占用
docker stats --no-stream
# 预期：各容器 CPU < 80%，MEM < 限制的 80%
```

### 2.2 日志查看

【从 compose volumes + 标准命令】：

```bash
# 查看某服务最近 100 行日志
docker compose logs --tail=100 {服务名}

# 实时跟踪日志
docker compose logs -f {服务名}

# 查看错误日志（过滤 ERROR/Exception/5xx）
docker compose logs {服务名} --since=24h | grep -iE 'error|exception|5[0-9][0-9]'

# Nginx 访问日志（统计 HTTP 状态码）
docker exec {nginx容器} tail -1000 /var/log/nginx/access.log | awk '{print $9}' | sort | uniq -c | sort -rn
```

### 2.3 日志清理

【标准内容】日志盘满会导致服务异常，定期清理：

```bash
# 清理 7 天前的应用日志
find /opt/{系统名}/logs -type f -name "*.log" -mtime +7 -delete

# 清理 Docker 容器日志（Docker 默认 json-driver 不自动轮转）
docker compose logs --tail=0 -f {服务名} > /dev/null &  # 先清空当前 buffer
# 或修改 /etc/docker/daemon.json 配置 log rotation：
# {
#   "log-driver": "json-file",
#   "log-opts": { "max-size": "100m", "max-file": "5" }
# }

# 清理 Docker 未使用的镜像/容器/卷（谨慎，先 dry-run 查看）
docker system df
docker system prune -v --filter "until=168h"  # 保留最近 7 天
```

⚠️ `docker system prune` 会删除未运行的容器和未使用的镜像，执行前用 `docker system df` 确认范围。

### 3.2 MySQL 备份

【标准内容 - 按数据库类型选一节，删除其他】：

#### MySQL 备份

```bash
# 全量逻辑备份（推荐，跨版本兼容）
docker exec {mysql容器} mysqldump \
  -u root -p"$DB_ROOT_PASSWORD" \
  --single-transaction \
  --routines \
  --triggers \
  --all-databases \
  | gzip > /opt/backup/mysql-$(date +%Y%m%d-%H%M%S).sql.gz

# 验证备份（解压检查）
gunzip -t /opt/backup/mysql-latest.sql.gz && echo "备份文件完整"

# 物理备份（XtraBackup，大库更快，需安装 percona-xtrabackup）
# docker exec {mysql容器} xtrabackup --backup --target-dir=/backup/$(date +%Y%m%d) ...
```

VERIFY: 备份文件大小 > 0；`gunzip -t` 退出码 0。

#### PostgreSQL 备份（如适用）

```bash
# 自定义格式备份（支持并行恢复）
docker exec {pg容器} pg_dump \
  -U $PG_USER \
  --format=custom \
  --compress=9 \
  $PG_DATABASE \
  > /opt/backup/pg-$(date +%Y%m%d-%H%M%S).dump

# 验证
docker exec {pg容器} pg_restore --list /opt/backup/pg-latest.dump | head
```

### 3.3 恢复

【标准内容 + 警告】：

⚠️ **恢复操作会覆盖现有数据，务必先在测试环境验证备份有效性，再在生产执行。**

#### MySQL 恢复

```bash
# 1. 停止业务服务（保留数据库运行）
docker compose stop {后端服务}

# 2. 恢复数据库
gunzip < /opt/backup/mysql-20260101-020000.sql.gz | \
  docker exec -i {mysql容器} mysql -u root -p"$DB_ROOT_PASSWORD"

# 3. 验证数据
docker exec {mysql容器} mysql -u root -p"$DB_ROOT_PASSWORD" -e \
  "SELECT COUNT(*) FROM {关键表};"

# 4. 重启业务服务
docker compose start {后端服务}
```

VERIFY: 关键表行数与备份时一致；应用功能正常。

#### PostgreSQL 恢复（如适用）

```bash
docker compose stop {后端服务}
docker exec -i {pg容器} pg_restore \
  -U $PG_USER \
  --clean --if-exists \
  -d $PG_DATABASE \
  < /opt/backup/pg-20260101-020000.dump
docker compose start {后端服务}
```

### 4.2 性能优化

#### MySQL 优化

```sql
-- 查看慢查询（需开启 slow_query_log）
SELECT * FROM mysql.slow_log ORDER BY start_time DESC LIMIT 10;

-- 查看表状态（数据量、引擎、行数）
SHOW TABLE STATUS FROM {数据库};

-- 分析表（更新索引统计信息）
ANALYZE TABLE {表名};

-- 优化表（回收已删除空间，会锁表，低峰期执行）
OPTIMIZE TABLE {表名};

-- 查看正在执行的查询
SHOW PROCESSLIST;

-- 杀掉长时间运行的查询（找到对应 Id 后）
KILL {process_id};
```

⚠️ `OPTIMIZE TABLE` 会锁表，大表执行可能导致业务阻塞，务必在低峰期执行。

#### PostgreSQL 优化（如适用）

```sql
-- 查看膨胀严重的表
SELECT relname, n_live_tup, n_dead_tup,
       ROUND(n_dead_tup::numeric / NULLIF(n_live_tup, 0) * 100, 1) AS dead_pct
FROM pg_stat_user_tables
WHERE n_live_tup > 1000
ORDER BY dead_pct DESC LIMIT 10;

-- 清理膨胀（VACUUM ANALYZE，不锁表）
VACUUM ANALYZE {表名};

-- 重建索引（CONCURRENTLY 不锁表，但耗时更长）
REINDEX INDEX CONCURRENTLY {索引名};
```

## 五、常见故障处理

> **agent 提示**：本章节**优先采纳历史 SOP 的真实故障经验**（高价值）。历史 SOP 缺失或不足时，用以下标准内容按 stack 类型补全。生成时删除本提示。

### 5.1 服务无法启动

| 症状 | 可能原因 | 诊断 | 修复 |
|------|---------|------|------|
| 容器立即退出（Exited） | `.env` 变量缺失 | `docker logs {容器}` 看启动错误 | 补全 `.env` 必填项，`docker compose up -d` 重启 |
| 容器反复重启 | 依赖服务未就绪 | `docker compose ps` 看 depends_on 状态 | 先确保 MySQL/Redis 启动成功，再启动后端 |
| 端口冲突 | 80/3306 等被占用 | `netstat -tlnp \| grep :80` | 停止占用进程或修改 compose 端口映射 |

### 5.2 数据库故障

【标准内容 - 按 database 类型保留对应小节】：

#### MySQL 故障

| 症状 | 可能原因 | 诊断 | 修复 |
|------|---------|------|------|
| 连接数超限 | `max_connections` 不足 | `SHOW STATUS LIKE 'Threads_connected';` 对比 `SHOW VARIABLES LIKE 'max_connections';` | 临时：`SET GLOBAL max_connections = 200;`；长期：修改 my.cnf |
| 慢查询导致业务卡顿 | 缺索引或大表全扫 | `SHOW PROCESSLIST;` 找 Time 大的查询 | `KILL {pid};`；长期：`EXPLAIN` 分析加索引 |
| 主从延迟（如有） | 大事务或网络 | `SHOW SLAVE STATUS\G` 看 Seconds_Behind_Master | 等待追平；检查网络；排查大事务 |
| 磁盘满 | binlog/data 膨胀 | `df -h`；`SHOW VARIABLES LIKE 'log_bin%';` | 清理 binlog：`PURGE BINARY LOGS BEFORE NOW() - INTERVAL 3 DAY;` |

#### PostgreSQL 故障（如适用）

| 症状 | 可能原因 | 诊断 | 修复 |
|------|---------|------|------|
| 锁等待 | 长事务持锁 | `SELECT * FROM pg_locks WHERE NOT granted;` | `SELECT pg_terminate_backend({pid});` |
| 连接超限 | `max_connections` 不足 | `SELECT count(*) FROM pg_stat_activity;` | 重启连接池；调大 max_connections |

### 5.3 缓存故障（Redis）

| 症状 | 可能原因 | 诊断 | 修复 |
|------|---------|------|------|
| OOM（内存满） | 未配置淘汰策略或大 key | `docker exec {redis容器} redis-cli INFO memory` | 配置 `maxmemory-policy`；排查大 key：`redis-cli --bigkeys` |
| 连接满 | 客户端未释放连接 | `redis-cli INFO clients` 对比 `maxclients` | 重启后端服务释放连接池 |
| 持久化失败 | 磁盘满或权限 | `redis-cli LATESTLOG` | 清理磁盘；检查 RDB/AOF 文件权限 |

### 5.4 Nginx 故障

| 症状 | 可能原因 | 诊断 | 修复 |
|------|---------|------|------|
| 502 Bad Gateway | 后端服务未启动或崩溃 | `docker compose ps` 确认后端状态 | 重启后端：`docker compose restart {后端服务}` |
| 504 Gateway Timeout | 后端响应慢 | 查后端日志与数据库连接数 | 优化慢查询；调大 nginx `proxy_read_timeout` |
| 499 Client Closed | 客户端主动断开（多为正常） | 统计占比 | 通常可忽略；占比高时排查前端超时设置 |
| SSL 证书过期 | 证书到期未更新 | `openssl s_client -connect {域名}:443` | 更新证书后 `docker exec {nginx} nginx -s reload` |

### 5.5 磁盘故障

```bash
# 磁盘满排查
df -h
du -sh /opt/{系统名}/* | sort -rh | head

# 常见占用大户及清理：
# 1. Docker 镜像/容器残留
docker system df
docker image prune -a --filter "until=168h"  # 删除 7 天前未用的镜像

# 2. 日志文件

# 3. 数据库备份堆积

# 删除超过保留期的备份（参考 3.1 策略）
```

## 七、应急预案

### 7.1 数据库数据丢失

**严重程度：P0。立即响应。**

1. 立即停止业务写入（`docker compose stop {后端服务}`）
2. 确认是否有有效备份（`ls -lh /opt/backup/`）
3. 按 3.3 恢复流程执行（先在测试环境验证）
4. 恢复后数据校验（关键表行数、最新记录时间）
5. 通知业务方评估数据丢失范围

### 7.2 系统全部不可用

**严重程度：P0。**

1. 确认服务器状态（`docker compose ps`、`df -h`、`free -h`）
2. 若磁盘满：参考 5.5 清理
3. 若内存不足：`docker compose restart` 重启服务
4. 若服务器宕机：联系机房/云厂商重启，启动后 `docker compose up -d`
5. 持续不可用超过 30 分钟：升级到技术负责人

### 7.3 安全事件（疑似入侵/数据泄露）

【需提供公司安全响应流程。标准建议】：

1. 立即断网（保留内存证据）
2. 保留日志（`docker compose logs > /tmp/incident-logs.txt`）
3. 修改所有密码（数据库、Redis、JWT_SECRET、SSH）
4. 排查后门（`crontab -l`、`last`、异常进程）
5. 通知安全负责人与业务方
