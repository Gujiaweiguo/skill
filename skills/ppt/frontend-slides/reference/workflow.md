# Frontend Slides — 工作流参考

## 完整流程

### 场景 1：技术大会演讲（从 0 新建）

```
Step 1 — 主题聚焦
  ├─ 定义核心论点（一句话：这场演讲想让听众带走什么？）
  ├─ 确定目标受众（初级/中级/高级前端）
  └─ 确定时长（15min / 30min / 45min → 对应页数）

Step 2 — 内容规划
  ├─ Title + 自我介绍
  ├─ 背景/动机（为什么讲这个）
  ├─ 核心内容（3-5 个技术要点，渐进式展开）
  │   ├─ 概念 → 代码 → 效果/数据
  │   └─ 每个要点 1-3 页
  ├─ 最佳实践/踩坑总结
  └─ Q&A + 资源链接

Step 3 — 生成
  ├─ 选择深色或浅色方案
  ├─ 代码块 → 等宽字体 + 行高亮
  ├─ 架构图 → 组件方块 + 箭头
  ├─ 数据 → 大数字卡片
  └─ 封面视觉（imageGen）

Step 4 — 验收（关注演讲可读性）
```

### 场景 2：内部分享/培训

```
Step 1 — 知识拆解
  ├─ 将主题拆为 4-8 个概念模块
  ├─ 每个模块 1-2 页
  └─ 确定需要代码演示的部分

Step 2 — 内容填充
  ├─ 概念页：一句大白话定义 + 要点展开
  ├─ 代码页：代码块 ≤ 25 行 + 关键行标注
  └─ 每页尾部可放"思考点"或"提问"

Step 3 — 生成
  ├─ 选择浅色/暖色方案（低视觉疲劳）
  ├─ 字体稍大（培训场景可读性优先）
  └─ 适当增加"回顾"页

Step 4 — 验收
```

### 场景 3：架构评审/技术方案

```
Step 1 — 确定评审范围
  ├─ 当前架构概况
  ├─ 痛点/问题
  ├─ 方案 A / B / C 对比
  └─ 推荐方案 + 理由

Step 2 — 内容生成
  ├─ 架构图页（组件关系、数据流）
  ├─ 对比表（方案 × 维度）
  ├─ 数据支持（性能测试结果）
  └─ 实施路径 / 迁移计划

Step 3 — 验收（关注逻辑完整性）
```

## 内容类型速查

### 每页类型选择

| 想传达 | 用哪种页面 | 核心元素 |
|--------|-----------|---------|
| 一个新概念 | Concept | 一句话定义 + 要点展开 |
| 一段代码 | Code | 代码块 + 行高亮 + 说明 |
| 架构设计 | Architecture | 方块图 + 箭头 + 组件说明 |
| 性能指标 | Data | 大数字卡片 + 趋势说明 |
| 选哪个方案 | Comparison | 对比表 + 推荐标注 |
| 学到了什么 | Best Practice | 要点列表 + 代码片段 + 反例 |

### 技术演讲每页容量参考

```
Title          1-2 句        最大最小
Agenda         4-6 项 + 时长   清单
Concept        1 定义 + 4-6 要点  一行定义
Code           ≤ 25 行代码    关键行 < 5
Architecture   4-8 个组件     组件名 ≤ 8 字
Data           3-6 个指标     每个指标 1 个大数字
Comparison     4-8 行 × 2-3 列  对比维度一致
Best Practice  3-5 条         每条 < 15 字
QA             N/A            联系方式 + 3-5 链接
```

## 常见检查清单

### 代码展示
- [ ] 代码块使用等宽字体
- [ ] 代码 ≤ 25 行，超过则截断或分段
- [ ] 关键行有视觉高亮（标记/颜色）
- [ ] 代码上方或下方有简要说明文字
- [ ] 语言/框架名称有标注

### 技术内容
- [ ] 技术术语准确（大小写、拼写正确）
- [ ] 版本号标注（Node 18, React 18 等）
- [ ] 性能数据有来源或测试环境说明
- [ ] 对比维度公正（不 biased）

### 演讲体验
- [ ] 不贴满屏文字
- [ ] 关键信息可 3 秒内获取
- [ ] 深色模式不混用饱和度过高的颜色
- [ ] 无动画过度依赖（PptxGenJS 动画受限，用递进页代替）
