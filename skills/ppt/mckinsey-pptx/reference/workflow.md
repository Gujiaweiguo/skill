# McKinsey PPT — 工作流参考

## 完整流程

### 场景 1：从 0 新建咨询 Deck

```
Step 1 — 需求澄清
  ├─ 明确核心问题（客户要什么决策？）
  ├─ 确定受众角色（C-level / 中层 / 投资方）
  └─ 确定页数范围（通常 8-15 页）

Step 2 — 叙事搭建（SCQA）
  ├─ Situation: 现状描述
  ├─ Complication: 关键挑战
  ├─ Question: 需要回答的问题
  └─ Answer: 核心结论/建议

Step 3 — 页面规划
  ├─ Title → Exec Summary → TOC
  ├─ Situation (2-3页) → Complication (2-3页)
  ├─ Analysis (3-5页) → Recommendation (2-3页)
  └─ Appendix (可选)

Step 4 — 内容填充
  ├─ 每页标题写为结论句
  ├─ MECE 分解内容要点
  ├─ 数据→图表，用 accent 高亮关键值
  └─ 每页标注数据来源

Step 5 — 设计生成
  ├─ 选择配色方案
  ├─ 按页面类型选择模板
  ├─ 生成 .pptx（pptxgenjs）
  └─ 补充封面视觉（imageGen）

Step 6 — QA 验收
  ├─ 麦肯锡三核心检查
  ├─ 标题即结论检查
  ├─ 视觉一致性检查
  └─ 数据来源完整检查
```

### 场景 2：将已有 PPT 改为麦肯锡风格

```
Step 1 — 读取分析（markitdown）
  ├─ 提取现有内容和结构
  └─ 评估现有页面类型

Step 2 — 内容重构
  ├─ 将描述性标题改为结论性标题
  ├─ 按 MECE 重组要点
  └─ 补充缺失的数据来源

Step 3 — 视觉重构
  ├─ 添加 McKinsey Header 栏
  ├─ 应用所选配色方案
  ├─ 标准化版式系统
  └─ 整理来源和页码

Step 4 — QA 验收
```

### 场景 3：研究报告/数据转演示

```
Step 1 — 提取核心发现（3-5 个关键洞察）
Step 2 — 按金字塔结构组织
  ├─ 结论先行（Executive Summary）
  ├─ 每个发现一页，按重要性排序
  └─ 每页标题写为洞察结论
Step 3 — 设计关键图表
Step 4 — 生成 + 验收
```

## 标题即结论 — 改写规则

| 原始标题（描述型） | 改写后（结论型） |
|------------------|-----------------|
| 市场规模分析 | 全球市场2026年将达$1.2万亿，年复合增长15% |
| 竞品定价策略 | 竞品A低价策略抢占市场，但利润率仅为行业一半 |
| 用户调研结果 | 68%用户因操作复杂放弃使用，简化流程为第一优先级 |
| 技术方案对比 | 云原生方案初始成本高30%，但3年TCO低45% |
| 实施路线图 | 分三阶段12个月完成，Q3实现首期上线 |

## 常见错误检查清单

### 结构问题
- [ ] 是否所有页面围绕一个核心结论？
- [ ] Executive Summary 是否覆盖了 deck 中的所有关键发现？
- [ ] 每个论点是否 MECE 分解（不重叠、无遗漏）？
- [ ] 叙事是否流畅（S → C → Q → A）？
- [ ] 建议是否源于前面的分析（不自创结论）？

### 标题问题
- [ ] 每页标题是否包含动词或判断（不是名词短语）？
- [ ] 标题能否独立传达该页信息（不看正文也能懂）？
- [ ] 标题是否数据驱动（包含数字或可验证的论断）？
- [ ] 标题是否在 3 秒内可读完？

### 视觉问题
- [ ] Header 栏是否统一（颜色、位置、字体）？
- [ ] 颜色是否严格限定在 5 色以内？
- [ ] 正文是否使用正确字号（12-14pt）？
- [ ] 是否有非必要的 3D 效果或装饰？
- [ ] 页码和来源位置是否每页一致？

## 交付物清单

- `presentation.pptx` — 最终 .pptx 文件
- `slides/*.js` — 每页生成脚本
- `slides/compile.js` — 编排脚本
- `slides/output/` — 输出目录
- 封面视觉素材（imageGen 生成）
- `mckinsey-check.md` — QA 检查结果
