# strategy-brief-generator Troubleshooting

本文件记录战略分析过程中的非显而易见问题和诊断流程。

## 已知问题

### 明源模板归纳冲突

**症状**：多份明源战略模板的章节结构不一致，归纳时不知道以哪个为准。

**原因**：明源不同时期的战略资料可能用不同框架（如有的用"五看"、有的用"四看"、有的是自由结构）。

**处理**：

1. 以"四看 → 定战略 → 执行计划"为主线（用户明确要求）。
2. 其他模板里有价值的维度，作为四看的补充子维度。
3. 在 `parsed/methodology/明源模板清单.md` 里记录每份模板的差异。
4. 不要强行合并互相矛盾的方法论；标注冲突，让用户决策。

### 竞品资料不完整导致能力矩阵出现空格

**症状**：某家竞品（如安盛祺/丰泽/科传）资料很少，能力矩阵大量空白。

**原因**：小众竞品公开资料少，可能只有一份 PPT 或官网截图。

**处理**：

1. 空白格标注"资料不足"，不要填"未发现"（未发现 ≠ 不存在）。
2. 用 web search 补充公开信息。
3. 仍然不足的，在 `review/` 里记录"需补充 XX 竞品资料"。
4. 不要因为资料少就低估竞品——小众竞品可能在某个垂直领域很强。

### 源码能力盘点过浅

**症状**：商管/AI 源码盘点只到模块名，没有能力细节。

**原因**：源码量大，第一版只做模块级盘点是合理的。

**处理**：

1. 第一版接受模块级盘点，在 `parsed/self/` 标注盘点深度。
2. 商管可复用已有 PRD（`$LANLNK_BASE/out/prd/商管系统/output/产品PRD.md`）补充能力细节。
3. AI 源码（lnkchatbi/langchat）盘点 README、路由、主要 service 类即可。
4. 不要为了"盘点完整"而陷入源码细节——战略分析不需要字段级精度。

### 市场搜索结果质量参差

**症状**：web search 返回的市场趋势资料质量不一，有的权威、有的是营销软文。

**原因**：商业地产 + AI 领域的公开研究报告质量差异大。

**处理**：

1. 优先采信：行业头部机构报告（克而瑞/赢商/戴德梁行/仲量联行）、政策文件、上市公司财报。
2. 谨慎采信：厂商白皮书、媒体软文、知乎/公众号观点。
3. 不采信：无来源的"行业共识"、AI 生成的趋势文章。
4. 每条市场信号标注来源类型（权威/一般/待验证）。

### SOP 资料覆盖不全

**症状**：商业地产 SOP 只覆盖部分业务领域（如只有招商和合同，没有物业和会员）。

**原因**：SOP 资料通常来自单一客户或单一项目，覆盖面有限。

**处理**：

1. 有什么 SOP 分析什么，不要伪造缺失部分。
2. 缺失部分标注"待补充 XX 领域 SOP"。
3. 可以用竞品资料中的业务流程描述作为 SOP 的补充证据（标注来源为竞品而非客户 SOP）。

### 战略判断变成竞品功能对比

**症状**：战略简报变成"竞品 A 有功能 X，蓝联也应该做功能 X"。

**原因**：这是战略分析最常见的陷阱——把竞品对标当成战略。

**处理**：

1. 竞品功能对比是"看竞对"的输入，不是"定战略"的结论。
2. 定战略要回答"蓝联应该做什么"，不是"竞品做了什么"。
3. 每个"应该做"的建议，必须连接到四看中的至少两个维度（如：市场需要 + 蓝联能做）。
4. 单纯"竞品有所以也要有"的建议，标注为低置信度假设。

### AI 场景推荐过度泛化

**症状**：战略简报里到处是"用 AI 提升效率"、"用 RAG 实现智能问答"。

**原因**：没有基于 SOP 推导具体场景，泛泛而谈。

**处理**：

1. 每个 AI 场景推荐必须连接到具体 SOP 流程或具体业务痛点。
2. 用 `SOP到AI问数场景映射.md` 和 `SOP到RAG知识场景映射.md` 作为依据。
3. 没有场景支撑的 AI 建议，标注为"待验证假设"。
4. 区分 AI问数 / RAG / AI助手 / AI决策，不要混为一谈。

### 第 3 章需求表和第 4 章竞对表不匹配

**症状**：第 3 章已经改成“岗位 + 业务场景”，但第 4 章仍然使用旧的“招商漏斗周报、经营问数、SOP知识问答”等需求点，导致无法逐行对比蓝联响应和竞对覆盖。

**原因**：需求视角重构后，只更新了需求章节，没有同步更新自身响应表和逐竞对矩阵。

**处理**：

1. 第 3 章需求表用 `岗位 | 业务场景 | 系统需求 | 商管需求 | 会员需求 | AI增强需求`。
2. 第 4 章自身响应表和竞对表的前两列也必须是 `岗位 | 业务场景`。
3. 对商管断言：`3.2 行集 == 4.2.1 行集 == 4.2.2 行集`。
4. 对会员断言：`3.3 行集 == 4.3.1 行集 == 4.3.2 行集`。
5. 改完后用脚本抽取表格前两列做顺序比对，不要只肉眼检查。

### B/D 路线混成一套打法

**症状**：报告同时说“中端已有系统 B 路线”和“低端无系统 D 路线”，但销售打法、周期、产品包和风险仍混在一起。

**原因**：客户分层只写了规模差异，没有写购买逻辑差异。中端客户买的是“AI 让已有系统产生价值”；低端/中小样板客户买的是“AI + 底座直接看到经营结果”。

**处理**：

1. 摘要必须前置“两条路线一句话”。
2. B 路线写清楚：不换系统、只读接入、简单 30 天/复杂 60 天交付验证、降本提效增收。
3. D 路线写清楚：不是传统系统铺量，而是选择性样板；先让铺位/合同/收费/销售/会员/活动/停车成为 AI 可用数据底座。
4. 同步检查行动计划、产品包、风险和已确认决策，避免“长尾暂不做”与“低端客户铺量”冲突。

### 内部汇报措辞割裂蓝联与蓝联创新

**症状**：报告大量写“蓝联创新应……”，听众却是蓝联合伙人和销售，容易被理解为“另一个团队自己的事”。

**原因**：没有根据听众调整组织主语。

**处理**：

1. 对蓝联合伙人/销售汇报时，标题用“蓝联商业地产 AI 战略分析总报告”。
2. 主语优先用“蓝联”，必要时写“蓝联（创新业务）”。
3. 蓝联创新业务定位为“商管 + AI 新增长线”，蓝联科技会员业务定位为“既有资产、客户关系、数据源和可选组件”。
4. 不写“蓝联创新独立于蓝联科技”，改写为“组织协同”。

### 交付周期和团队配置旧口径残留

**症状**：用户已明确“AI coding 不需要产品经理、简单 30 天复杂 60 天”，报告仍残留“产品经理”“90 天试点”“3-6 周/3-6 月”。

**原因**：行动计划、摘要、销售话术、证据台账等章节分散，局部替换容易漏。

**处理**：

1. 团队配置统一为：`AI工程师 + 实施/AgentOps + 销售`。
2. 周期统一为：`简单 30 天，复杂 60 天` 或 `30-60 天交付验证`。
3. 全文检索并清理：`产品经理`、`90天`、`90 天`、`3-6 周`、`3-6 月`。
4. 若引用历史材料名称含“90天”，改成资料概括名，避免销售口径混乱。

## 诊断流程

### 战略简报质量自检

生成战略简报后，按以下清单自检：

```text
□ 管理层摘要能否一句话讲清战略定位？
□ 四看的每个维度是否都有证据支撑？
□ 看自己是否诚实面对了短板？
□ 定战略是否收敛到 2-3 个重点？
□ 执行计划是否有可验证的阶段目标？
□ AI 场景推荐是否都连接到了具体 SOP？
□ 证据台账是否完整（每条判断可追溯）？
□ 是否有未标注的假设（混在判断里）？
□ 是否有越界内容（PRD 字段、客户方案叙事、报价）？
```

### 资料覆盖度评估

分析开始前，评估资料覆盖度：

```text
资料覆盖度 = {
  methodology: 有几份明源模板？结构是否完整？
  market: 本地市场资料有多少？需要联网补多少？
  sop: 覆盖了哪些业务领域？缺失哪些？
  competitors_商管: 5 家竞品资料量如何？是否有完整方案？
  competitors_会员: 3 家竞品资料量如何？
  self_商管: 源码是否可达？PRD 是否已生成？
  self_会员: materials 资料是否完整？
  self_AI: 源码是否可达？
}
```

覆盖度低于 60% 的维度，在 `review/` 记录并提示用户补充资料。
