# Project Proposal Generator — 立项报告生成 Agent Pipeline

基于 `需求解析 + 竞品对标 + 素材匹配 + 标准结构编排 + 文档输出` 的方案。
用户说"XX客户要做XX，帮我立项"，Agent 负责调研对标、匹配素材、生成报告。

---

## 配置读取

设置 `LANLNK_BASE` 环境变量指向素材根目录（默认当前目录），然后设置路径变量：

| 变量 | 路径 |
|------|------|
| `$MATERIALS_DIR` | `$LANLNK_BASE/materials` |
| `$PROPOSALS_DIR` | `$LANLNK_BASE/materials/14-proposals`（立项 + 方案统一存放） |
| `$CASES_DIR` | `$MATERIALS_DIR/04-cases` |
| `$PRODUCTS_DIR` | `$MATERIALS_DIR/03-products` |
| `$COMPANY_DIR` | `$MATERIALS_DIR/01-company-overview` |

---

## 业务流程

```
客户需求（谁 + 做什么 + 为什么现在做）
        ↓
P0: 需求解析 → 提取关键参数
        ↓
P1: 竞品对标调研 → WebSearch 搜集信息 + 用户提供名单
        ↓
P2: 素材匹配 → 从素材库检索相关产品方案和案例
        ↓
P3: 内容编排 → 按立项报告标准结构组织内容
        ↓
P4: 文档生成 → 输出草稿 + Word/PPT 内容包（YAML/Markdown）
        ↓
P5: 输出核验 → 检查完整性，给出报告说明
```

## P0: 需求解析

### 0.1 从用户描述中提取关键参数

| 参数 | 说明 | 示例 |
|------|------|------|
| 项目名称 | 客户项目全称 | 福州正祥广场会员系统项目 |
| 客户单位 | 客户公司全称 | 福州正祥商业管理有限公司 |
| 客户行业 | 所属行业 | 购物中心、商业地产、零售 |
| 客户规模 | 面积、商户数、年客流 | 30万㎡、500+商户、年客流2000万 |
| 业务需求 | 要做什么 | 会员系统、私域商城、智慧停车 |
| 痛点描述 | 为什么要做 | 会员分散、数据不通、营销手段单一 |
| 对标对象 | 想参考谁 | 福州本地头部购物中心（用户提供或自动搜索） |
| 预算范围 | 大概预算（可选） | 80-120万 |
| 决策背景 | 管理层关注什么 | ROI、竞品压力、集团数字化战略 |
| 汇报对象 | 谁来拍板 | 总经理 / 分管领导 / 业务负责人 |

### 0.2 信息不完全的处理

用户可能只说"正祥广场想上会员系统"，此时 Agent 应主动追问补齐关键信息：

```
✅ 必须补齐：
  - 项目名称（立项必须有名称）
  - 客户行业（决定对标方向和产品匹配）
  - 业务需求（决定推荐哪些产品方案）
  - 汇报对象（决定叙事深度和篇幅）
  
⚠️ 建议补齐：
  - 客户规模（面积/商户数，影响方案规模）
  - 痛点描述（影响立项必要性的说服力）
  
💡 可以搜索补齐：
  - 对标对象（如果用户没给，自动搜索同城/同业态标杆项目）
  - 客户背景信息（天眼查/企查查风格的基本信息）
```

## P1: 竞品对标调研

### 1.1 对标调研策略

```
用户是否提供对标名单？
    ├── 是 → 优先调研用户指定的对标对象
    └── 否 → 自动搜索：
              ├── 同城同类购物中心（同城市、同业态）
              ├── 同规模标杆（同体量、同定位）
              └── 行业头部（全国知名项目）
```

### 1.2 调研内容框架

对标每个竞品，搜集以下维度的信息：

| 维度 | 调研内容 | 搜索关键词示例 |
|------|---------|---------------|
| 会员体系 | 会员等级、积分规则、权益 | "XX广场 会员等级"、"XX商城 会员权益" |
| 数字化工具 | 小程序、APP、停车系统 | "XX广场 小程序"、"XX商场 智慧停车" |
| 营销玩法 | 社群运营、直播、私域 | "XX购物中心 私域运营"、"XX商场 社群" |
| 业绩效果 | 会员数、复购率、GMV | "XX购物中心 会员数"、"XX商场 客流" |
| 痛点不足 | 用户投诉、体验差评 | "XX广场 停车难"、"XX商城 小程序不好用" |

### 1.3 对标分析报告输出

Agent 搜索后整理为结构化对标报告：

```markdown
## 对标分析：福州头部购物中心会员体系

### ① 福州仓山万达广场（竞品A）
- 会员等级：普卡-银卡-金卡-钻石，消费累积升级
- 小程序功能：停车缴费、积分兑换、优惠券、店铺导航
- 停车策略：消费满100免停2小时
- 会员规模：约80万+（据公开报道）
- 差距/不足：积分通用性差，跨业态打通不足

### ② 福州东百中心（竞品B）
- 会员等级：东百会员PLUS体系
- 特色：市区核心商圈，历史文化街区+现代商业
- 活动：高频主题活动+社群运营
- 数据：会员消费占比超60%（据公开资料）

### ③ 福州万象城（竞品C）
- 定位：高端商业
- 会员服务：一对一客服、专属停车位、生日礼
- 数字化：华润通统一会员平台
```

## P2: 素材匹配

### 2.1 匹配素材类型

| 素材类型 | 用途 | 检索范围 |
|---------|------|---------|
| 产品方案 | 建设方案推荐 | `$PRODUCTS_DIR/` 按 domain 匹配 |
| 客户案例 | 效果证明 | `$CASES_DIR/` 按 industry/scenario 匹配 |
| 公司概况 | 公司资质背书 | `$COMPANY_DIR/` 必选 |
| 资质荣誉 | 实力证明 | `$MATERIALS_DIR/02-qualifications/` 必选 |
| 实施方法论 | 建设路径 | `$MATERIALS_DIR/05-implementation/` 可选 |
| 服务体系 | 售后保障 | `$MATERIALS_DIR/06-services/` 可选 |
| 人员团队 | 执行团队 | `$MATERIALS_DIR/07-personnel/` 可选 |

### 2.2 匹配策略

```
1. 按业务需求匹配产品方案
   → 如"会员系统"匹配：会员CRM、SCRM、小程序、统一会员平台
  
2. 按行业+场景匹配案例
   → 如"购物中心"+"会员营销"匹配：时尚天河、世欧广场、宝能…
   → 优先选同行业 ✅ 完整案例，再选 ⚠️ 部分完整案例
  
3. 必选素材
   → 公司简介 + 资质证照（都是通用背书）
```

### 3.2 篇幅策略

| 篇幅 | 适用场景 | 内容裁剪 |
|------|---------|---------|
| ✅ **简要版**（8-12页PPT） | 初次沟通、快速立项 | 封面+摘要+背景+必要性+对标+方案概览+预算+建议 |
| ✅ **标准版**（20-30页PPT / 8-12页Word） | 正式立项申请 | 完整9章节 |
| ✅ **详细版**（40+页PPT / 15+页Word） | 重大投资、高层汇报 | 完整9章节 + 详细对标数据 + 多方案对比 |

### 3.3 商管/经营类方案的特殊结构调整

> **2026-07-05 新增**。适用于商业地产商管 ERP、经营管理平台、资管系统等"经营管控类"立项。

当方案主体是商管/经营类时，标准 9 章节的以下部分建议调整：

| 章节 | 标准结构 | 商管/经营类调整 | 依据 |
|---|---|---|---|
| **2.3 核心痛点** | 泛泛描述 | **按岗位拆痛点**（招商总/财务总/营运总/企划总/物业总/IT总各自的痛） | 15 号矩阵 |
| **6.2 核心功能模块** | 按模块列功能（资源/招商/合同…） | **改为按岗位组织**（每个岗位的病与药），或岗位为主+模块为辅 | 11 号 §8.3 |
| 痛点开篇 | 功能语言（"合同到期漏""应收乱"） | **决策层三问**：租没租出去？能收多少？收到多少？ | 11 号 §8.3 |
| 证明/证据 | 统一引用案例 | **证明口径分级**：Z1 实证/Z2 竞品行业对标/Z3 蓝联产品状态/Z4 待试点验证；没有 Z1 不硬凑 | 15 号矩阵 §1.3 |

**岗位病药数据来源**：`$LANLNK_BASE/knowledge/sales/methodology/15-商业地产岗位病药矩阵.md`（7 岗位 × 痛点/场景/功能/价值/证明完整矩阵）。

**结构参考**：`$PROPOSALS_DIR/商管系统方案汇报/蓝联商管ERP系统方案汇报_客户版.pptx`（v2 岗位病药结构，18 页：P3 决策层三问 → P4 经营驾驶舱 → P5 多项目对标 → P6-P11 六岗位病药 → P15 能力证明分级 → P16 模块全景作为能力地图）。

> **关键原则**：岗位视角优先于模块视角。客户方按岗位组织，按模块展示客户"找不到自己的页"；按岗位组织，每个决策人看到自己的病与药。模块全景不放开篇（避免产品自嗨），放在能力证明段作为补充。

## P4: 文档生成

### 4.1 生成策略

> **核心原则**：立项报告 Skill 只处理“要不要做、为什么做、怎么立项”这类内部决策内容。
> 它和 company-intro-generator 是**独立并行 Skill**，不共享对外介绍叙事模板。
>
> | Skill | 生成什么 | 输出目录 |
> |-------|---------|---------|
> | project-proposal-generator | 立项建议书（要不要做、为什么做、怎么决策） | `$PROPOSALS_DIR/{项目}/` |
> | company-intro-generator | 公司介绍/方案书（我们有什么、做过什么） | `$PROPOSALS_DIR/{项目}/` |
>
> **Word**：输出结构化内容包（`.word-content.md`）→ word-master 排版生成 `.docx`。
> **PPT**：输出立项专用内容包 → 立项模板编译生成 `.pptx`。
> 两种格式走不同引擎，业务 Skill 只负责"说什么"。

### 4.2 生成流程

```
P3 编排完成 → 得到完整的内容大纲和素材引用
    ↓
4.2.1 先生成 Markdown 草稿（供审阅确认）
    ↓ 用户确认后（或默认继续）
4.2.2 输出结构化 Word 内容包 → 调用 word-master 生成 Word 文档
    ↓ 或（可选）
4.2.3 输出结构化 PPT 内容包（YAML, `*_proposal-pptx.yaml`） → 调用 ppt-master 生成 PPT
    ↓
4.2.4 输出到 `$PROPOSALS_DIR/{项目名称}/`
```

### 4.3 Markdown 草稿

- 输出到 `$PROPOSALS_DIR/{项目名称}/立项建议书_草稿.md`
- 包含完整的 9 章节内容和所有数据
- 供用户审阅确认，确认后转正式文档

### 4.4 Word 文档：输出结构化 Word 内容包

**不再使用 python-docx 自行生成 Word。** 改为按 [Word 内容包规范](../../word/word-master/reference/word-content-package-spec.md) 输出 `.word-content.md`，交给 word-master 做专业排版。

**流程：**
1. 内容编排完成后，按 Word 内容包规范生成 `.word-content.md`
2. 放入 `$PROPOSALS_DIR/{项目名称}/content-packages/{项目名称}_立项建议书_{YYYYMMDD}.word-content.md`
3. 告知用户："内容已编排完成，正在调用 word-master 排版生成正式文档..."
4. word-master 读取内容包后自动生成 `.docx`，输出到 `$PROPOSALS_DIR/{项目名称}/`

**立项报告章节 → Word 内容包映射：**

| 立项章节 | 内容包元素 | 说明 |
|---------|-----------|------|
| 封面 | `cover:` 元数据 | 标题+副标题+日期+机密标记 |
| 目录 | 自动生成 | word-master 根据 heading-1 自动生成 |
| 管理层摘要 | `heading-1` 章节 | 分页 |
| 项目背景 | `heading-1` + `heading-2` 小节 | 分页开始 |
| 对标分析 | `heading-1` + `heading-2` + `table: comparison` | 对比表格 |
| 建设方案 | `heading-1` + `heading-2` + `table: function-matrix` | 功能清单表 |
| 投入预算 | `heading-1` + `table: pricing-table` | 预算明细表 |
| 风险与对策 | `heading-1` + bullet list | 分页 |
| 建议与下一步 | `heading-1` | 不分页 |
| 附录 | `heading-1` + 公司资质/案例引用 | 不分页 |

**输出文件：**
- Word 内容包：`$PROPOSALS_DIR/{项目名称}/content-packages/{项目名称}_立项建议书_{YYYYMMDD}.word-content.md`
- Word 文档：`$PROPOSALS_DIR/{项目名称}/立项建议书_{项目简称}_{YYYYMMDD}.docx`（由 word-master 生成）

### 4.5 PPT 生成

立项 PPT 使用项目专用内容包生成，保持和 Word 内容包一致的立项叙事，不引入公司介绍模板的章节顺序。PPT 只承担“立项说明”，不承担“公司介绍/投标述标”。

### 4.6 对标调研报告

- 作为附录输出到 `$PROPOSALS_DIR/{项目名称}/对标调研报告.md`
- 包含所有搜索到的对标项目详细信息和数据来源

## P5: 输出核验

### 5.1 完整性检查

- 9 个章节是否都有内容（至少简要版覆盖所有章节）
- 对标分析是否有 ≥2 个对标项目
- 方案是否有 ≥1 个产品推荐
- 预算是否有分项（软件、实施、服务、运维）

### 5.2 数据一致性检查

- 项目名称在全文是否统一
- 客户信息是否一致
- 预算数字是否前后匹配

### 5.3 输出报告

## 与兄弟 Skill 的关系

```
material-importer（素材入库）
        ↓
project-proposal-generator（立项报告） ← 你现在在这里
        ↓ (客户确认立项后，可选)
company-intro-generator（方案汇报/公司介绍）
        ↓ (如进入招标阶段，可选)
bid-doc-master（投标文件）
```

- 立项报告在业务流程最前端，先于方案汇报
- 立项报告与后续方案材料是可复用关系，不是默认联动关系
- 立项报告生成的调研信息可作为后续方案汇报的参考输入
