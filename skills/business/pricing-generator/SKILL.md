# Pricing Generator — 报价单生成 Agent Pipeline

## DocSpec 质量基线

本 skill 生成的报价单、功能报价清单、方案对比和第三方集成说明必须遵守 `/opt/code/skill/references/docspec/`，重点执行 `DocSpec-通用文档质量规范.md`、`PPT与Word内容包质量规范.md` 和 `文档验收清单.md`。定价来源、税率、公式、优惠空位、第三方边界和待确认项必须显式保留。

基于 `产品功能清单 + 标准定价 + 二开清单 + 第三方集成 + 方案对比` 的方案。
用户说"帮我出个报价"，Agent 负责整理功能清单、套用定价、组合二开和第三方、输出可面客报价单。

| `$PRD_DIR` | `$LANLNK_BASE/prd` | 产品功能清单来源 |

| 模板 | 路径 | 说明 |
|---|---|---|
| SAAS 模板 | `$REFERENCES_DIR/报价模板_SAAS.md` | 首年含租用+实施，次年纯租用 |

## 业务流程

```
客户名 + 产品 + 定价 + 模式（+ 二开清单可选）
        ↓
P0: 输入解析 → 确认产品/模式/定价/二开清单
        ↓
P1: 功能清单整理 → 按模块组织产品功能（报价支撑）
        ↓
P2: 标准产品定价 → 软件许可/租用 + 实施 + 售后
        ↓
P3: 定制开发（可选）→ 二开项 × 单价，单独作为可选项
        ↓
P4: 第三方集成（另行计费）→ 列出不报价
        ↓
P5: 输出报价单 → 标准产品 + 可选二开 + 方案对比 + 服务说明
```

## P0: 输入解析

### 0.1 从用户描述中提取关键参数

| 参数 | 说明 | 示例 |
|------|------|------|
| 客户名 | 用于输出目录命名 | XX资产公司 |
| 产品 | 报价的产品（**单个或组合**）| MI / CRM / AI Skills / MI+CRM / MI+CRM+AI Skills |
| 报价模式 | SAAS 或 私有化 | SAAS |
| 标准定价 | 产品的标准定价（用户提供或从模板取）| 首年5万/次年2万 |
| 二开清单（可选）| 来自 requirement-evaluator | 41 人天 / 2 项 |
| 客户规模（可选）| 影响档位 | 1 项目 / 写字楼+商业 |

### 0.2 定价来源确认

定价数据是报价的核心。按优先级确认：

1. **用户提供**（最准）：用户直接告知"首年5万次年2万"
2. **报价模板取**（标准产品）：读 `references/报价模板_<模式>.md`，如 CRM 有标准定价
3. **已有报价单推断**（参考）：如同产品已有其他客户报价单，参考其定价
4. **都不存在**：提示用户提供定价"未找到 <产品> 的标准定价，请提供首年/次年费用，或参考竞对报价估算"

### 0.3 需求评估前置（自动衔接 requirement-evaluator）

报价依赖需求评估的二开清单。**自动检查，无需用户手动分两步**：

```
检查 $PROPOSALS_DIR/<客户>/需求评估_*.md 是否存在？
  ├── 存在 → 直接读取二开清单，进入 P1
  └── 不存在 → 自动先执行 requirement-evaluator 流程：
           需求转化 → 结构化 → 产品匹配(7类分类) → 二开识别 → 成本估算 → 输出评估报告
           → 再继续 pricing-generator P1
```

**用户体验**：只需说"给这个需求出报价"，skill 自动完成 **评估 → 报价** 一站式。

> 多产品组合时，需求评估对每个产品分别评估（或一份综合评估含多个产品维度），二开清单按产品归集。

### 0.4 多产品组合报价规则

当产品参数含多个（如 MI+CRM+AI Skills）时：

| 环节 | 单产品 | 多产品组合 |
|---|---|---|
| 需求评估 | 评估一个产品 | 各产品分别评估（或综合评估），二开按产品归集 |
| 功能清单（P1）| 一份功能清单 | **按产品分段合并**（Part 1A: MI 模块 / Part 1B: CRM 模块 / Part 1C: AI Skills 岗位）|
| 定价（P2）| 套一个模板 | **各产品分别套模板，合计**（MI 5万 + CRM 7.2万 + AI 3万 = 15.2万）|
| 报价单（P5）| 一份报价 | **分产品 Section + 合计 + 组合方案对比** |
| 方案对比 | A/B/C 二开组合 | **叠加产品组合维度**（纯 MI / MI+CRM / MI+CRM+AI）|

**组合方案对比示例**（6 种主要场景）：
```
方案 A：纯 MI             首年 5万    次年 2万
方案 B：纯 CRM            首年 7.2万  次年 3万
方案 C：纯 AI Skills(3岗)  首年 3万    次年 2万
方案 D：MI + CRM          首年 12.2万 次年 5万
方案 E：MI + AI(3岗)      首年 8万    次年 4万
方案 F：MI+CRM+AI(3岗)    首年 15.2万 次年 7万
```
> 其他组合（如 CRM+AI Skills）按需组合，各产品定价相加。

### 0.4 信息不完全的处理

```
✅ 必须确认：
  - 客户名（输出目录）
  - 产品（单个或组合：MI / CRM / AI Skills / MI+CRM / MI+CRM+AI Skills）
  - 报价模式（SAAS/私有化）
  - 标准定价（首年/次年费用）

⚠️ 建议确认：
  - 二开清单（是否接 requirement-evaluator）
  - 客户规模（影响档位：首项目/新增项目）
  - 二开单价（默认 1800 元/天，可调整）
```

## P1: 功能清单整理（按模块组织）

### 1.1 为什么需要功能清单

报价单不只是数字，还要让客户知道"花这笔钱买到什么"。功能清单是报价单的**内容说明**支撑，按模块组织让客户清晰看到产品价值。

### 1.2 整理方法

读产品功能清单（345 项 flat 列表），按业务模块归类：

**模块划分原则**：
- 按业务域归类（招商/资源/合同/计费/运营/物业/报表/移动端/系统管理）
- 每个模块概括核心能力（不列每一项功能，而是写一段描述）
- 参考 CRM 报价模板的"内容说明"列风格

**输出格式**（每个模块一段）：
```markdown
### 模块 N：XXX
功能项1、功能项2、功能项3、...（用顿号串联核心功能，让客户看到覆盖面）
```

### 1.3 参考：MI 商管系统 9 大模块

| 模块 | 核心能力 |
|---|---|
| 资源管理 | 铺位/空间/楼层/多经/车位/租控可视化 |
| 招商管理 | 品牌库/招商计划/绩效/渠道/客户管理 |
| 合同管理 | 全生命周期/多租金模式/变更/终止/清算 |
| 财务管理 | 出账/应收/收费/对账/发票/税务 |
| 运营管理 | 销售/商户/装修/巡检/推广 |
| 物业服务 | 工单/报修/工程条件/设备 |
| 报表看板 | 仪表盘/报表设计器/Excel/可视化分析 |
| 移动端 | 管理端+租户端 |
| 系统管理 | 组织/权限/工作流/审批 |

> 其他产品（CRM/AI Skills）按各自功能清单整理对应模块。

### 1.4 参考：CRM 会员营销系统模块

| 模块 | 核心能力 |
|---|---|
| 会员管理 | 电子会员/会员卡券/积分管理/标签/等级/资产 |

| 数据分析 | 会员画像/消费分析/流失预警 |
| 消息中心 | 短信/模板消息/推送通知 |
| 移动端 | 会员小程序/电子会员卡/积分商城 |
| 系统管理 | 组织/权限/门店/数据隔离 |

## P2: 标准产品定价

### 2.1 报价结构（4 大类，复用现有模板体系）

| 类别 | 费用项 | 性质 | 说明 |
|---|---|---|---|
| **一、软件核心模块** | 平台许可/租用 | 一次性（私有化）或年费（SAAS）| 产品标准定价 |
| **二、开发交付服务** | 集成/定制/二开 | 一次性 | 可选，见 P3 |
| **三、实施服务** | 启动/设计/上线/验收 | 一次性（首年）| 含 N 人天 |
| **四、售后/运营服务** | 年费 | 按产品类型区分（见下方说明）|

> **⚠️ 第四类性质按产品类型区分（关键认知）**：
>
> | 产品类型 | 第四类称呼 | 性质 | 典型内容 | 措辞要求 |
> |---|---|---|---|---|
> | **MI / CRM**（软件产品）| 售后服务 | 被动维护 | bug修复/安全更新/版本升级 | "年度售后服务"/"维护费" |
> | **AI Skills**（运营服务）| **运营服务** | **主动运营** | 周复盘/命中率调优/知识库更新/效果看板/SLA保障/持续优化 | **"年度运营服务"/"运营服务费"** |
>
> **关键认知**：AI 岗位 Skill 卖的是**运营服务**，不是软件许可。蓝联团队持续运营，确保 Skill 越用越准——成果导向，不是交付后不管的工具。生成 AI Skills 报价单时，措辞**必须用"运营服务"**，不能用"售后/维护"。

### 2.2 SAAS 模式定价拆分

```
首年费用 = 软件租用费 + 实施费（一次性）
次年费用 = 软件租用费（纯年费，售后含在内）

示例（MI）：
  软件租用 20,000/年 + 实施 30,000（首年）= 首年 50,000
  次年 = 20,000（纯租用）
```

### 2.3 私有化模式定价拆分

```
首年费用 = 软件终生授权 + 实施费（一次性）
次年费用 = 年售后费（可选，通常按合同 10%）

示例（CRM 私有化，参考模板）：
  软件授权 45,000 + 实施 18,000 + 智慧商圈 20,000 = 首年 ~90,000
  次年 = 售后 9,000（可选）
```

### 2.4 客户规模档位

参考报价模板的"首项目 / 新增项目"双列：
- **首项目**：客户首个项目上线，含完整实施
- **新增项目**：客户已有项目，新增项目实施费减免（或免费）

## P3: 定制开发（可选，单独报价）

### 3.1 二开项来源

从 requirement-evaluator 的"本产品二开清单（仅 🔨 项）"提取。

每项包含：
- 二开项名称 + 需求来源
- 复杂度（S/M/L）+ 人天
- 报价 = 人天 × 单价

### 3.2 二开报价计算

```
二开单价 = 1,800 元/人天（标准报价单价，可在运行时调整）
单项报价 = 人天 × 单价

示例：
  线索确权规则：11 天 × 1,800 = 19,800
  房屋售卖全流程：30 天 × 1,800 = 54,000
```

> **注意**：requirement-evaluator 用 1,500 元/天作为**成本基准**估算；pricing-generator 用 1,800 元/天作为**报价单价**。差额是利润。

### 3.3 二开作为可选项

二开不混入标准产品报价，而是**单独列为可选项**，让客户按需选择：
- 标准产品（必选）= 基础方案
- + 二开项 1（可选）= 增强方案
- + 二开项 2（可选）= 完整方案

### 3.4 自动衔接 requirement-evaluator（--devkit）

提供 `--devkit <评估报告路径>` 时，脚本自动解析需求评估报告中的二开清单，
覆盖 optional_items：

```bash
uv run python generate_quote.py --customer XX --product MI --mode SAAS \
    --devkit $PROPOSALS_DIR/XX/需求评估_MI_20260705.md
```

解析规则：
- 从评估报告的「二、二开清单」表格提取二开项（名称/复杂度/人天）
- 报价 = 人天 × 1,800（pricing-generator 报价单价）
- S/M 级标记 P0，L 级标记 P1
- 覆盖原有 optional_items

## P4: 第三方集成（另行计费，不报价）

### 4.1 列出但不报价

需求书涉及的第三方集成项，在报价单里**列明但不报价**（等客户指定供应商后单独计费）：

| 集成类型 | 说明 | 计费方式 |
|---|---|---|
| 支付网关 | 微信/支付宝/对公在线缴费 | 客户指定后单独计费 |
| 电子签 | 法大大/e签宝等 | 按第三方报价 + 集成人天 |
| 客流系统 | AI 客流摄像头数据接入 | 按客流系统报价 + 接口对接 |
| 其他蓝联产品 | langchat/LnkChatBI 等 | 另产品报价 |

### 4.2 为什么不报价

- 第三方费用取决于客户选择的供应商（价格差异大）
- 蓝联只收集成对接的人天费（待范围明确后算）
- 提前报价容易偏差，不如列明"另行计费"

## P5: 输出报价单

### 5.1 输出目录与命名

```
$PROPOSALS_DIR/<客户名>/
├── 需求评估_<产品>_<日期>.md          ← requirement-evaluator 产物（前置）
├── <产品>_报价功能清单及报价单_<模式>_<日期>.md  ← 旧版 Markdown 产物（保留作底稿）
├── 报价单_<产品>_<模式>_<客户>_<日期>.xlsx       ← 本 Skill 主产物（Excel）
└── （后续）方案汇报/投标文件...
```

**Excel 命名规则**：`报价单_<产品>_<模式>_<客户>_<YYYYMMDD>.xlsx`

**生成命令**（在 skill 目录下）：

```bash
cd skills/business/pricing-generator
uv sync                                                       # 首次或依赖更新后
uv run python generate_quote.py \
    --customer <客户名> --product <MI|CRM|AI> \
    --mode <SAAS|私有化> --date <YYYYMMDD> \
    [--positions <N>]     # AI 专用：岗位数（默认3，范围2-6）
    [--devkit <path>]     # 可选：需求评估报告路径，自动解析二开清单覆盖定制开发项
```

> **产品支持**：MI ✅ / CRM ✅ / AI ✅（三产品均已实现）/ 私有化模式待补充定价 ⏳

       一、标准产品报价（必选）              ← 浅蓝小节标题

### 5.3 Excel 格式规范（参考正祥报价单）

为保持面客报价单的视觉一致性，所有 Excel 输出遵循以下格式规范（参考 `14-proposals/正祥会员系统/报价单_SAAS_正祥_20260618.xlsx`）：

| 元素 | 规范 | 说明 |
|---|---|---|
| 字体 | 微软雅黑（全表统一）| 中文商业文档标准 |
| 标题行 | 14pt 粗体，A:F 合并居中 | 跨 6 列 |
| 信息行 | 9pt 灰字（555555），A:F 合并左对齐 | 服务商/客户/日期/有效期 |
| 表头行 | **深蓝填充 001E5A8A** + 白字 10pt 粗体，居中 | 边框 thin |
| 数据行 | 10pt 正常，内容说明左对齐，其余居中 | 自动换行，行高按内容估算 |
| 小节标题 | **浅蓝填充 00E4F0FA** + 11pt 粗体，A:F 合并左对齐 | 「一、二、三…」分类 |
| 汇总/合计行 | **浅红填充 00F9E5DD** + 11pt 粗体，金额橙色 D46B43 | 突出报价数字 |
| 金额格式 | `#,##0`（千分位）| 20,000 / 50,000 |
| 列宽（报价单 Sheet）| A=8 / B=22 / C=50 / D=14 / E=14 / F=18 | 序号/名称/说明/首年/次年/备注 |

## 报价模型汇总

### 定价公式

```
【标准产品】
SAAS：
  首年 = 软件租用费 + 实施费
  次年 = 软件租用费

私有化：
  首年 = 软件终生授权 + 实施费
  次年 = 年售后费（可选，约合同 10%）

【定制开发】
  单项 = 人天 × 1,800 元/天
  总计 = Σ 各项
```

### 利润逻辑

```
成本基准（requirement-evaluator 用）：1,500 元/人天
报价单价（pricing-generator 用）：1,800 元/人天
利润率 = (1800-1500)/1800 ≈ 17%（二开部分）

标准产品定价由用户提供/模板取，利润率内含（不额外加价）
```

## 与兄弟 Skill 的关系

```
requirement-evaluator（需求评估 + 二开成本）
    ↓ 输出二开清单 + 成本估算
pricing-generator（报价单）← 你在这里
    ↓ 输出报价单
company-intro-generator（方案汇报，引用报价）
    ↓
bid-doc-master（投标商务标，引用报价）
```

**上下游衔接**：
- **上游**：requirement-evaluator 的二开清单 → pricing-generator 的定制开发部分
- **下游**：pricing-generator 的报价单 → company-intro-generator 方案汇报引用 / bid-doc-master 商务标引用

**复用关系**：
- 报价单的"功能清单"可复用到方案汇报的"产品介绍"
- 报价单的"方案对比"可复用到售前沟通
- 报价单的"第三方集成清单"可复用到实施计划
