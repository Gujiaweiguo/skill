# 素材质量标准参考

此文件定义 8 类素材的质量标准框架和评分规则，由 material-importer 的 P3 阶段引用。
仅在需要质量评估时加载此文件。

---

## 01-company-overview — 公司概况（6 层必答）

```
第 1 层：WHO WE ARE — 公司身份
  ✅ 全称、成立时间、注册资本、总部地址
  ✅ 团队规模（研发/实施/销售）
  ✅ 企业性质
  ✅ 一句话定位（60 字以内，差异化）

第 2 层：WHAT WE DO — 业务全景
  ✅ 核心业务线（每线：定位、目标客户、产品形态）
  ✅ 服务模式（SaaS年费/私有化部署/联合运营）
  ✅ 业务收入占比（可选）

第 3 层：WHERE WE ARE — 行业地位
  ✅ 生态伙伴（腾讯/阿里/抖音/金蝶 等）
  ✅ 资质背书（高企/双软/ISO27001/等保/邓白氏）
  ✅ 协会身份

第 4 层：WHO WE SERVE — 客户覆盖
  ✅ 服务客户总数、覆盖行业
     ⚠️ 常见缺失：需补充
  ✅ 标杆客户 LOGO 墙（带图片）
  ✅ 按行业分类的客户列表
  ✅ 典型客户续约率/满意度

第 5 层：OUR STRENGTHS — 核心优势
     ⚠️ 常见缺失：需补充
  ✅ 技术优势（自研、专利、架构）
  ✅ 行业经验（深耕 X 年、案例数）
  ✅ 服务优势（响应速度、驻场能力）
  ✅ 生态优势（对接微信支付/支付宝/停车系统等）
  ✅ 每个优势有具体数据或案例支撑

第 6 层：DEVELOPMENT HISTORY — 发展历程
     ⚠️ 常见缺失：需补充
  ✅ 成立至今关键里程碑（年份+事件）
  ✅ 产品迭代路线
  ✅ 团队扩张节点
  ✅ 重要客户签约时间线

质量红线：
  ❌ 只有空洞口号无数据支撑 → 拒绝
  ⚠️ 缺第 4/5/6 层 → 标记，提示补充
```

## 02-qualifications — 资质证照（一证一文件）

```
每份证照必须包含：
  ✅ 证书名称（精确全称）
  ✅ 发证机构
  ✅ 发证日期（issued）
  ✅ 有效期至（expires）
  ✅ 证书编号
  ✅ 证书图片（从原始文档提取）
  ✅ 适用场景（投标/展示/信息安全证明等）

质量红线：
  ❌ 只有"公司通过了XX认证"一句话 → 拒绝
  ⚠️ 缺有效期或编号 → 标记，提示补充
  ⚠️ 缺证书图片 → 标记，提示提供扫描件
```

## 03-products — 产品方案

```
每份产品方案必须包含：
  ✅ 产品定位（解决什么问题）
  ✅ 核心功能列表
  ✅ 目标客户/适用场景
  ✅ 技术架构/产品蓝图（含图片）
  ✅ 差异化优势（为什么比别的强）
  ✅ 服务模式（SaaS/私有化/联合运营等）

质量红线：
  ❌ 只有功能堆砌，没有定位和场景 → 拒绝
  ❌ 有趣描述无功能 → 拒绝
```

## 04-cases — 客户案例（6 模块一客户一文件）

```
每个案例必须包含：
  ✅ ① 项目背景 — 客户是谁、行业地位、面临什么痛点
  ✅ ② 项目需求 — 客户要解决什么问题、期望目标
  ✅ ③ 实现功能 — 交付了什么系统、功能清单
  ✅ ④ 实现效果 — 量化结果（增长数据、效率提升）
  ✅ ⑤ 项目图片 — 至少 1 张：系统截图/现场照片/方案架构图
  ✅ ⑥ 合同信息 — 服务期限、合同金额（可选）、服务内容

质量红线：
  ⚠️ ④ 无量化数据 → 弱案例，标记
  ⚠️ ⑤ 无图片 → 纯文本案例，标记
  ⚠️ ⑥ 无服务期限 → 标记，提示补充
  ❌ ①②③ 任一项缺失 → 拒绝
```

## 05-implementation — 实施方法论（8 层完整体系）

```
第 1 层：实施方法论总纲
  ✅ 方法论名称与核心思想
  ✅ 三层架构：策略层 → 操作层 → 支持层
  ✅ 总体原则：整体规划，分步实施
  ✅ 方法论示意图

第 2 层：实施阶段详解（4 阶段 × 每阶段含任务、交付物、里程碑）
  ✅ 阶段 1：项目准备（立项分析、组织建立、启动大会）
  ✅ 阶段 2：蓝图设计（业务调研、流程梳理、静态数据准备）
  ✅ 阶段 3：系统实现（定制开发、数据导入、试运行、用户培训）
  ✅ 阶段 4：验收交付（正式上线、运行跟踪、项目验收）

第 3 层：项目进度计划模板
  ✅ 三阶段划分（试运行上线 → 正式上线推广 → 跟踪服务）
  ✅ 每阶段：时间规划、建设目标、验收标准
  ✅ 甘特图/进度计划图

第 4 层：项目组织架构
  ✅ 客户方组织：领导小组 → 应用组 → 系统管理组 → 关键用户组
  ✅ 我方组织：领导小组 → 实施组 → 咨询组 → 开发组 → 服务支持组
  ✅ 各角色 RACI 矩阵
  ✅ 人员投入保障比例

第 5 层：质量管理体系
  ✅ 质量标准体系（范围/周期/工作内容）
  ✅ 沟通管理机制（周例会制度、会议纪要流程）
  ✅ 文档管理规范（备份策略、归档规则）
  ✅ 问题管理机制（分级处理流程）
  ✅ 变更管理流程

第 6 层：开发流程与工具
     ⚠️ 常见缺失：需补充
  ✅ 开发模式（瀑布/敏捷/混合）
  ✅ 代码管理（Git 分支策略）
  ✅ CI/CD 流水线
  ✅ 测试体系（单元/集成/性能/安全）
  ✅ 技术栈说明

第 7 层：文档交付清单
  ✅ 需求规格说明书、上线验收确认单、项目调研材料
  ✅ 系统设计文档、API 接口文档、运维手册、用户手册
     ⚠️ 常见缺失：需补充

第 8 层：风险管控
     ⚠️ 常见缺失：需补充
  ✅ 常见风险识别（需求变更/人员变动/技术风险/进度风险）
  ✅ 风险应对预案
  ✅ 应急响应流程

质量红线：
  ❌ 方法论没有阶段划分 → 拒绝
  ⚠️ 缺第 6/7/8 层 → 标记，提示补充
```

## 06-services — 服务体系（9 层完整框架）

```
第 1 层：服务理念与承诺
  ✅ 服务理念
  ✅ 服务承诺（可用性、故障响应时间）
  ✅ 服务体系介绍

第 2 层：SLA 服务等级协议
  ✅ 问题分级：一级（系统崩溃）/ 二级（功能异常）/ 三级（一般问题）
  ✅ 各级响应时间和解决时间
  ✅ 服务方式：电子支持 7×24 / 热线 7×24 / 远程 7×24 / 现场
  ✅ 节假日加急保障

第 3 层：运维服务内容
  ✅ 日常运维（资料维护、积分处理、活动运营、权益维护）
  ✅ 技术支持（故障排查、性能优化、备份恢复、安全修复）
  ✅ 培训与指导（操作培训、新功能培训、手册/视频/文档）

第 4 层：服务团队架构
     ⚠️ 常见缺失：需补充
  ✅ 一线支持（客服/运维）→ 二线支持（研发）→ 三线支持（专家）
  ✅ 团队规模与资质
  ✅ 排班机制

第 5 层：数据安全与灾备
     ⚠️ 常见缺失：需补充
  ✅ 数据加密方案
  ✅ 访问控制策略
  ✅ 定期备份策略（频率/保留周期）
  ✅ 灾难恢复方案（RTO/RPO 目标）
  ✅ 安全审计机制

第 6 层：服务范围与边界
     ⚠️ 常见缺失：需补充
  ✅ 免费服务范围（标准运维含什么）
  ✅ 增值服务范围（运营陪跑/定制开发/活动策划）
  ✅ 不在服务范围的事项
  ✅ 服务升级机制

第 7 层：服务流程与工具
     ⚠️ 常见缺失：需补充
  ✅ 问题提报流程（在线客服→工单→升级→关闭）
  ✅ 工单系统
  ✅ 满意度调查机制
  ✅ 服务报告（月报/季报/年报）
  ✅ 知识库与 FAQ

第 8 层：联合运营服务
  ✅ 导流-策动-创意-技术-执行 一站式
  ✅ 运营策划支持
  ✅ 活动执行（线上到线下、公域到私域）
  ✅ 合作案例

第 9 层：技术转移与知识产权
     ⚠️ 常见缺失：需补充
  ✅ 源码交付策略
  ✅ 知识产权归属
  ✅ 技术转移方案

质量红线：
  ❌ 没有响应时间 → 拒绝
  ⚠️ 缺第 4/5/6/7/9 层 → 标记，提示补充
```

```
质量红线：
  ❌ 只有角色名无具体人员 → 拒绝
  ⚠️ 缺证书扫描件 → 标记，提示补充
  ⚠️ 缺社保证明 → 标记，提示补充
```

## 质量评分规则

| 评分 | 含义 | 处理方式 |
|------|------|---------|
| ⭐⭐⭐ | 完整，所有必填项齐全 | 直接入库 |
| ⭐⭐ | 部分缺失，核心内容完整 | 入库 + 标记 ⚠️ + 提示补充 |
| ⭐ | 严重缺失，核心内容不全 | 拒绝入库，说明原因 |
