# Safety & Ethics — 合规与凭据安全红线

本文件定义竞品分析的合规边界与凭据安全规则。**这些是红线，违反任一条都会导致 skill 拒绝执行或中止。**

## 法律与伦理底线

### 竞品情报采集的合法来源

| 来源 | 合法性 | 说明 |
|---|---|---|
| 竞品主动提供的测试账号 | ✅ 合法 | 竞品方授权访问 |
| 公开渠道获得的 demo 账号（官网注册、免费试用）| ✅ 合法 | 公开邀约 |
| 竞品公开发布的操作手册、产品视频、官方文档 | ✅ 合法 | 公开信息 |
| 竞品公开发布的方案 PPT、白皮书、行业洞察 | ✅ 合法 | 公开信息（但作弱证据）|
| 行业展会、公开 webinar 上的演示 | ✅ 合法 | 公开活动 |
| 客户主动分享的竞品方案/招标响应 | ⚠️ 灰区 | 需确认客户有权分享，且不违反 NDA |
| G2/Capterra/Trustpilot 公开评测 | ✅ 合法 | 公开信息 |
| 招标公告/中标公示 | ✅ 合法 | 公开信息 |

### 禁止的采集行为

| 行为 | 状态 | 原因 |
|---|---|---|
| 伪造身份注册竞品试用 | ❌ 禁止 | 欺骗，且易被识破毁誉 |
| 盗用他人账号登录竞品 | ❌ 禁止 | 违法 |
| 绕过验证码/CAPTCHA | ❌ 禁止 | 违反 ToS，可能违法 |
| 绕过权限控制访问未授权页面 | ❌ 禁止 | 越权访问 |
| 绕过反爬虫机制批量抓取 | ❌ 禁止 | 违反 ToS |
| 漏洞扫描/SQL 注入/压力测试 | ❌ 禁止 | 攻击行为，违法 |
| 批量导出竞品客户数据 | ❌ 禁止 | 侵犯隐私，违法 |
| 假装客户预约竞品 demo 套取信息 | ❌ 禁止 | 欺骗 |
| 通过竞品员工私下套取内部资料 | ❌ 禁止 | 违反竞业/保密 |
| 窃取竞品源代码 | ❌ 禁止 | 违法 |

### skill 内置合规检查（P0 强制）

Agent 在 P0 阶段必须确认：

1. **demo 账号来源**：用户必须说明账号来源（竞品方提供/公开注册/客户分享）。来源不明时拒绝执行 demo 模式。
2. **ToS 冲突标记**：如目标站点 ToS 明确禁止自动化访问，skill 标记冲突并提示用户咨询法务，不自动执行。
3. **凭据安全承诺**：用户必须确认理解凭据只进 `.auth.json`，不进 git/日志/报告。
4. **最小采集承诺**：用户必须确认只采集能力分析必需的页面，不抓取客户数据。

任一项未确认，skill 进入阻塞状态并提示：

```
合规检查未通过：<具体原因>
请确认：
1. demo 账号来源合法（竞品方授权或公开渠道）
2. 不绕过任何访问控制
3. 凭据将只写入 .auth.json（gitignored）
4. 只采集能力分析必需数据
```

## 凭据安全

### .auth.json 规范

**位置**：`$ANALYSIS_ROOT/.auth.json`（即 `$LANLNK_BASE/out/prd/商管系统/competitor-analysis/<vendor>/.auth.json`）

**权限**：mode 0600（仅所有者可读写）

**格式**：

```json
{
  "username": "llkj",
  "password": "llkj123",
  "source": "竞品方提供测试账号",
  "obtained_at": "2026-07-06",
  "account_role_observed": "管理员",
  "notes": "账号有效期未知，建议复核"
}
```

**强制规则**：

- 首次写入时打印：`"请确保 .auth.json 已加入 .gitignore：$LANLNK_BASE/out/prd/商管系统/competitor-analysis/**/.auth.json"`
- 二次运行复用，不重复询问
- `.gitignore` 必须覆盖：在 `$LANLNK_BASE/.gitignore` 或仓库根 `.gitignore` 添加 `**/competitor-analysis/**/.auth.json`
- **禁止**：把凭据写入 SKILL.md、references、报告、日志、截图文件名、git commit message

### 截图脱敏

截图在写入 `imgs/` 前必须脱敏（规则见 `evidence-ledger.md` 的脱敏章节）。

**额外要求**：

- 截图文件名不含客户名/手机号：用 `合同管理_3_新合同申请表单.png`，不用 `张三合同.png`
- 截图 EXIF/metadata 必须清除（playwright 截图默认无 EXIF，但需确认）
- 含客户列表的截图整体打码或只保留表头

### 报告脱敏

`lanlnk-product-improvement-recommendations.md` 与 `evidence-ledger.md` 中：

- 不出现真实客户名（替换为"客户A"）
- 不出现真实手机号/身份证号/邮箱
- 不出现真实金额（替换为数量级）
- 截图引用必须指向已脱敏的 `imgs/` 文件

## 数据最小化

### 采集范围

demo 探测**只采集能力分析必需**的内容：

| 采集 | 说明 |
|---|---|
| 菜单结构 | ✅ 必采 |
| 页面路由 | ✅ 必采 |
| 表单字段名/placeholder | ✅ 必采 |
| 表格列标题 | ✅ 必采 |
| 按钮文本 | ✅ 必采 |
| 流程步骤（页面间跳转）| ✅ 必采 |
| 截图（每页 1-3 张关键状态）| ✅ 必采 |
| 客户/商户真实数据 | ❌ 不采 |
| 交易明细/财务数据 | ❌ 不采 |
| 个人隐私（手机/身份证/地址）| ❌ 不采 |
| 报表中的真实数字 | ❌ 不采（只采报表名和维度）|
| 后台日志/系统配置 | ❌ 不采 |

### 保留与删除

- 截图与分析产物保留在 `$ANALYSIS_ROOT/`（gitignored 或内部仓库）
- 原始凭据 `.auth.json` 在分析完成后建议删除（除非需要定期复核）
- raw 转换产物（markitdown 输出）由 material-importer 管理，本 skill 不删除
- 不把竞品截图/UI 文字用于客户面向材料

## ToS 冲突处理

当目标站点 ToS 明确禁止自动化访问时：

1. **P0 阶段检测**：Agent 读取目标站点 ToS（或询问用户是否已确认）
2. **标记冲突**：在 source-inventory.json 标记 `tos_conflict: true`
3. **不自动执行**：skill 进入阻塞，提示用户：
   ```
   目标站点 ToS 可能禁止自动化访问。
   建议：
   1. 咨询法务确认民事暴露
   2. 改用手工浏览 + 人工截图（不调用 Playwright）
   3. 或取得竞品方书面授权
   ```
4. **降级路径**：用户可选择 `--manual-screenshot` 模式，由用户手工截图后放入 `imgs/`，skill 只做能力分析

## 合规自查清单（每次运行前）

Agent 在 P0 必须打印并要求用户确认：

```
合规自查：
[ ] demo 账号来源合法（竞品方授权/公开渠道/客户有权分享）
[ ] 不绕过任何访问控制（验证码/权限/反爬）
[ ] 不做漏洞扫描或压力测试
[ ] 凭据将只写入 .auth.json（mode 0600，gitignored）
[ ] 只采集能力分析必需数据，不抓取客户隐私
[ ] 截图将脱敏后再写入证据
[ ] 产物不直接用于客户面向材料
```

任一项未确认，skill 不执行 demo 模式（manual 模式可正常执行）。

## 与 doc-generator 凭据规范的一致性

本 skill 的 `.auth.json` 规范复用 doc-generator 的 P2.5 阶段规则：

- 同样的文件位置约定（`_input/` 或分析根目录）
- 同样的 mode 0600 权限
- 同样的 gitignore 要求
- 同样的二次复用不重复询问

差异：

- doc-generator 的 `.auth.json` 在 `$USERGUIDE_BASE/_input/{name}/`
- 本 skill 的 `.auth.json` 在 `$ANALYSIS_ROOT/`（更靠近分析产物，便于按竞品隔离）
