# Requirement Evaluator — 需求满足度与二开成本评估 Agent Pipeline

## DocSpec 质量基线

本 skill 生成的需求满足度评估、覆盖矩阵、二开清单、成本估算和打单建议必须遵守 `/opt/code/skill/references/docspec/`，重点执行 `DocSpec-通用文档质量规范.md`、`方案与投标文档质量规范.md` 和 `文档验收清单.md`。覆盖率、二开范围、成本基准、报价基准必须分开表达。

基于 `需求解析 + 产品功能匹配 + 二开复杂度分级 + 成本估算 + 应标建议` 的方案。
用户说"评估一下这个需求"，Agent 负责转化需求、匹配产品、识别二开、算成本、给应标建议。

---

## 配置读取

设置 `LANLNK_BASE` 环境变量指向素材根目录（默认当前目录），然后设置路径变量：

| 变量 | 路径 | 说明 |
|------|------|------|
| `$MATERIALS_DIR` | `$LANLNK_BASE/materials` | 素材库根 |
| `$PROPOSALS_DIR` | `$MATERIALS_DIR/14-proposals` | 评估报告输出位置 |
| `$INCOMING_DIR` | `$LANLNK_BASE/incoming` | 需求文档暂存区 |
| `$PRD_DIR` | `$LANLNK_BASE/prd` | 产品 PRD 套件根（功能清单来源）|

### 产品功能清单定位

评估基准产品的功能清单由 product-prd-generator 生成，位于 `$PRD_DIR/<产品>/output/功能清单.md`。

| 产品代号 | 功能清单路径 | 说明 |
|---|---|---|
| MI / 商管系统 | `$PRD_DIR/商管系统/output/功能清单.md` | 345 项功能，含 existing/missing 状态 |
| CRM / 会员系统 | `$MATERIALS_DIR/03-products/CRM会员系统功能清单.md` | 会员 CRM 功能 |
| AI Skills | `$MATERIALS_DIR/11-cre-ai-skills/02_机会与产品/岗位 AI Skills 增强性与摩擦消除分析矩阵.md` | AI 岗位 Skill 能力 |

> 若功能清单不存在，Agent 应提示用户先运行 product-prd-generator 生成，或由用户提供功能清单路径。

## 业务流程

```
客户需求文档（docx/pdf/md）+ 评估基准产品
        ↓
P0: 输入解析 → 转化需求文档 + 定位功能清单
        ↓
P1: 需求结构化 → 把需求书拆成可评估的原子条目
        ↓
P2: 产品能力匹配 → 逐条对照功能清单，标注满足度
        ↓
P3: 二开项识别 → 从差距项提取二开清单 + 复杂度分级
        ↓
P4: 二开成本估算 → 人天 × 单价 + 分期归属
        ↓
P5: 输出评估报告 → 满足度 + 二开清单 + 成本 + 应标建议
```

## P0: 输入解析

### 0.1 从用户描述中提取关键参数

| 参数 | 说明 | 示例 |
|------|------|------|
| 需求文档路径 | 客户需求书的文件路径 | `incoming/需求-XX招商管理.docx` |
| 评估基准产品 | 评估对照哪个产品 | MI（商管系统）/ CRM（会员）/ AI Skills |
| 客户名（可选）| 用于输出目录命名 | XX资产公司 |
| 单价基准（可选）| 二开人天单价，默认 1500 元/天 | 1500 / 2000 |
| 客户业态（可选）| 辅助理解需求 | 写字楼出租 / 购物中心 / 产业园区 |

### 0.2 需求文档转化

若需求文档为 docx/pdf，先用 markitdown 转化为 md：

```bash
markitdown "<需求文档>" -o "<同目录>/<同名>.md"
```

转化后的 md 作为评估的权威输入。

### 0.3 定位产品功能清单

根据评估基准产品，定位 `$PRD_DIR/<产品>/output/功能清单.md`。

- 功能清单包含功能名、状态（existing/partial/missing）、置信度
- `existing` = 产品已有该功能
- `partial` = 部分实现
- `missing` = 未实现

若功能清单不存在，提示用户：
```
未找到 <产品> 的功能清单。请先运行 product-prd-generator 生成，或提供功能清单路径。
```

### 0.4 信息不完全的处理

用户可能只说"评估一下这个需求"，此时 Agent 应主动追问：

```
✅ 必须确认：
  - 需求文档路径（评估对象）
  - 评估基准产品（对照哪个产品）

⚠️ 建议确认：
  - 客户名（影响输出目录）
  - 客户业态（辅助理解需求语义）
```

## P1: 需求结构化

### 1.1 把需求书拆成可评估的原子条目

读转化后的需求 md，按需求书的章节结构，拆成**可独立评估的原子需求条目**。

**拆分原则**：
- 保留需求书的原始章节编号（一一对应可追溯）
- 每个条目是一个可评估的功能点（太笼统的拆开，太细的合并）
- 每条提取：需求描述 + 所属模块 + 优先级推断

### 1.2 原子条目结构

```markdown
| 编号 | 需求条目 | 所属模块 | 优先级推断 |
|---|---|---|---|
| 5.1.1 | 写字楼房源建档（房号/面积/产权/状态）| 房源资产管理 | 必需 |
| 5.1.2 | 商业标准商铺（编号/面积/业态/筛选）| 房源资产管理 | 必需 |
```

**优先级推断依据**：
- **必需**：需求书明确"必须/核心/本期建设"
- **期望**：需求书描述为建设目标但未强调必需
- **可选**：需求书提及但属扩展/远期

## P2: 产品能力匹配

### 2.1 满足度分类（7 类，通用）

> **通用性原则**：以下分类不绑定具体产品。评估任何基准产品（MI/CRM/langchat/LnkChatBI 等）时均适用。"本产品"指用户指定的评估基准产品。

| 标记 | 含义 | 判定依据 | 是否算二开 |
|---|---|---|---|
| ✅ | 本产品完全满足 | 功能清单 existing 且语义匹配 | 否 |
| ⚙️ | 本产品标准功能（配置即可）| 标准模块的配置项（预警规则/打印模板/审批流/计费规则/字段扩展等）| 否（配置）|
| 🔌 | 本产品标准 API（外部对接）| 本产品提供 API/接口，让外部系统主动对接 | 否（API 已有）|
| 🔨 | 本产品需二开 | 功能清单 missing，需开发新功能 | **是** |
| 🤖 | 其他蓝联产品承接 | 非本产品范围，蓝联另一产品能做 | 否（另一产品范围）|
| 🔗 | 第三方系统集成 | 行业惯例由第三方专业系统负责 | 否（集成费另算）|
| ❓ | 待讨论/待确认 | 需求不够明确，需客户确认规则后才能评估 | 待定 |
| ❌ | 技术不可行/无替代 | 当前技术无法可靠实现，或非任何产品范围 | 否（给降级方案）|

### 2.2 通用判断决策树

判断一个需求项属于哪一类，**按顺序排查**（不能跳步）：

```
Step 1: 功能清单 existing 且语义匹配？
    └── 是 → ✅ 完全满足

Step 2: existing 但需要配置/调规则？
    └── 是 → ⚙️ 标准功能配置（如预警阈值、审批流、计费规则、打印模板）
    └── 注意：很多需求看似"要开发"，实际是标准模块的配置项

Step 3: 本产品有 API/接口能力，需求是"让外部系统接"？
    └── 是 → 🔌 标准 API 对接（本产品提供 API，外部系统自己接）

Step 4: missing，但蓝联另一产品有此能力？
    └── 是 → 🤖 其他蓝联产品承接（标注哪个产品）
    └── 如：活动ROI归因 → langchat 工作流；数据问数 → LnkChatBI；会员积分 → CRM

Step 5: missing，但行业惯例是第三方专业系统做？
    └── 是 → 🔗 第三方集成（标注哪类第三方）
    └── 如：客流热力图 → 客流系统；支付 → 支付网关；电子签 → 法大大

Step 6: missing，需要本产品开发新功能？
    └── 是 → 🔨 需二开（进入 P3 复杂度分级）

Step 7: 需求描述不够明确/规则未定？
    └── 是 → ❓ 待讨论（标注待确认什么）

Step 8: 技术上难以可靠实现？
    └── 是 → ❌ 技术不可行（给降级方案）
```

> **关键原则**：不能把所有 missing 都直接判为 🔨 二开。必须先排查 Step 2-5（是否有标准配置/API/其他产品/第三方承接），最后才到 Step 6 判 🔨。这是评估准确性的核心。

### 2.3 替代方案要求（必填）

每个**非 ✅ 项**，必须给出替代方案，不能只标注"不满足"：

| 满足度 | 替代方案要求 | 示例 |
|---|---|---|
| ⚙️ | 说明"配置即可"，标注在哪个模块配置 | "预警模块配置 7天/15天阈值" |
| 🔌 | 说明"本产品提供 API，外部系统对接"，标注接口 | "MI 提供 external integration API，POS 自己接" |
| 🔨 | 标注二开复杂度 + 人天（见 P3）| "L 复杂，30 人天" |
| 🤖 | 标注由哪个蓝联产品承接 + 为什么归那个产品 | "langchat 活动复盘 Skill 承接（属 AI Skills 范围）" |
| 🔗 | 标注第三方系统类型 + 集成方式 + 是否单独计费 | "客流系统负责热力图，集成费另算" |
| ❓ | 标注待确认什么规则/维度，确认后才能评估 | "分级维度/阈值未定，待客户确认" |
| ❌ | 给降级方案（运营人工/简化版/分期后置）| "客流×坪效校验不可靠，建议运营人工核查异常" |

### 2.4 匹配输出表

```markdown
| 需求编号 | 需求条目 | 满足度 | 对应产品功能 | 替代方案 |
|---|---|---|---|---|
| 5.1.1 | 写字楼房源建档 | ✅ | 铺位管理、空间结构管理 | — |
| 5.4.4 | 活动 ROI 自动归因 | 🤖 | — | langchat 活动复盘 Skill 承接 |
| 6.3 | 客流热力图 | 🔗 | — | 客流系统（第三方）负责，MI 接收数据 |
| 6.8 | 防瞒报校验 | ❌ | — | 技术不可靠，建议运营人工核查 |
```

### 2.5 匹配原则

- **关联证据**：✅/⚙️/🔌 必须引用功能清单里的具体功能名，不能泛泛说"有"
- **替代方案必填**：非 ✅ 项必须给替代方案，不能只标"不满足"
- **通用不绑定产品**：判断规则适用于任何评估基准产品（MI/CRM/langchat/LnkChatBI），不写死具体产品名
- **宁可深挖不轻判二开**：missing 先排查配置/API/其他产品/第三方（Step 2-5），最后才判 🔨（Step 6）

## P3: 二开项识别 + 复杂度分级

### 3.1 从差距项提取二开清单

从 P2 的 🔨（需二开）项中，提取二开清单。

```markdown
| # | 二开项 | 需求来源 | 复杂度 | 说明 |
|---|---|---|---|---|
| 1 | 写字楼房屋买卖合同/产权交易 | 5.1.2 / 5.3 | L | 新业务线，含合同/房款/认购 |
| 2 | 销售数据 API 上报（对接 POS）| 5.6.2 | M | 接口对接 + 数据校验 |
```

### 3.2 复杂度分级标准

| 复杂度 | 特征 | 人天基准 | 典型场景 |
|---|---|---|---|
| **S 简单** | 配置项/规则/报表/字段扩展 | 2-5 天 | 加一个计费规则、一个审批流字段、一个报表列 |
| **M 中等** | 单模块新功能/接口对接/页面开发 | 8-15 天 | API 对接（POS/停车/财务）、佣金确权规则、商户画像分级 |
| **L 复杂** | 跨模块/新业务线/核心架构改动 | 20-40 天 | 房屋买卖全流程、活动ROI归因引擎、热力图可视化系统 |

**分级依据**：
- 是否跨模块（单模块 = S/M，跨模块 = L）
- 是否新业务线（新业务线 = L）
- 是否需要接口对接（对接 = M 起）
- 是否核心架构改动（架构改动 = L）

### 3.3 人天基准可调整

人天基准是默认参考值。skill 运行时若用户提供历史项目数据或自定义基准，优先使用用户值。

## P4: 二开成本估算

### 4.1 成本计算公式

```
单二开项成本 = 复杂度人天基准 × 单价
  - S: 取基准中值 3 天
  - M: 取基准中值 11 天
  - L: 取基准中值 30 天

总二开工作量 = Σ 各项人天
总二开成本 = Σ 各项成本 = 总人天 × 单价

应标报价建议 = 总二开成本 × 利润系数(1.3~1.8)
  - 满足度高(✅>70%)：利润系数 1.5~1.8（产品价值主导）
  - 满足度中(✅40-70%)：利润系数 1.3~1.5（二开主导，竞争报价）
  - 满足度低(✅<40%)：不建议主力应标，或重新定位
```

### 4.2 单价基准

| 默认 | 说明 |
|---|---|
| 1500 元/人天 | 蓝联标准开发人天单价（可在运行时由用户覆盖）|

### 4.3 分期归属

每个二开项标注建议分期：

| 分期 | 含义 | 判定 |
|---|---|---|
| **P0** | 首期必须 | 客户必需 + 经营闭环核心 |
| **P1** | 二期推进 | 客户期望 + 增强体验 |
| **P2** | 三期深化 | 可选 + 深度分析 |
| **P3** | 远期规划 | PRD 已规划该阶段 |

### 4.4 成本汇总表

```markdown
| 分期 | 二开项数 | 总人天 | 总成本(万元) |
|---|---|---|---|
| P0 | 3 | 45 | 6.75 |
| P1 | 2 | 22 | 3.30 |
| P2 | 1 | 15 | 2.25 |
| **合计** | **6** | **82** | **12.30** |
```

## P5: 输出评估报告

### 5.1 输出目录

```

├── 需求评估_<产品>_<YYYYMMDD>.md      ← 主报告

- 客户目录不存在则创建

**降级规则**：
- 若用户未提供客户名，报告输出到 `$INCOMING_DIR/需求评估_<产品>_<YYYYMMDD>.md`
- 并提示："后续确认客户名后，建议将报告迁入 `$PROPOSALS_DIR/<客户名>/`"

## 与兄弟 Skill 的关系

```
material-importer（素材入库）
    ↓
requirement-evaluator（需求评估）← 你在这里
    ↓ 评估通过，决定打单
project-proposal-generator（立项报告）
    ↓
company-intro-generator（方案汇报/公司介绍）
    ↓
bid-doc-master（投标文件）
```

**横向依赖**：
- **product-prd-generator** → 产出产品功能清单（本 Skill 的评估基准）
- 本 Skill 产出的评估报告 → 可被 project-proposal-generator / company-intro-generator 复用

**复用关系**：
- 评估报告里的"满足度分析"可复用到方案汇报的"需求理解"章节
- 评估报告里的"二开清单"可复用到报价单
- 评估报告里的"需客户确认问题"可复用到售前沟通清单

## 素材缺口提示

评估过程中若发现基准不足，主动提示：

| 场景 | 提示内容 |
|------|---------|
| 无产品功能清单 | "未找到 <产品> 的功能清单。请先运行 product-prd-generator 生成，或提供功能清单路径。" |
| 需求文档质量差 | "需求文档描述过于笼统，已做合理拆分，建议与客户确认需求细节后再定稿评估。" |
| 功能清单过时 | "功能清单生成日期较早，可能与当前产品状态有偏差，建议重新生成。" |
| 二开项识别不充分 | "部分需求项语义模糊，二开评估可能偏低，建议与开发团队复核。" |
